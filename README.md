# modu

> 고민할 시간에, 만나요.

**modu**는 업무 회의부터 친구 약속과 모임까지, 여러 사람이 만날 시간을 정할 때 **꼭 함께할 사람/함께하면 좋은 사람**이 참여하기 어려운 날짜·시간만 입력하면 **어떤 날을 먼저 보고, 어떤 날은 피하면 좋은지 설명하는 시간 판단 보조 도구**입니다. 로그인 없이 링크로 응답을 받고, 열람 조건이 충족되면 캘린더에서 전체 기간 기준 상대 추천도, 날짜별 판단 근거, 특정 날짜·시간의 가능 여부를 확인합니다. modu는 최종 시간을 자동으로 확정하지 않으며, 최종 시간은 이 판단 근거를 바탕으로 참여자들이 외부(Slack·카톡·구두 등)에서 정합니다. 일정을 만든 사람도 참여자 중 한 명이며, 별도 관리자 개념이나 관리자 페이지는 두지 않습니다.

이 저장소는 실제로 동료에게 링크를 보내 사용할 수 있는 **알파 버전**입니다.

## 설계 의도 요약

- **왜 결과 화면이 시나리오 데모인가** — 평가자가 6명의 역할을 나눠 상황을 재현하는 시간 비용을 줄이기 위해, 대표적인 회의 조율 상황을 빠르게 전환해 보는 평가 장치를 넣었습니다.
- **왜 '피하고 싶은 시간'을 별도 입력으로 받지 않는가** — 입력은 '회의가 어려운 날짜·시간' 하나로 단순화하고, 해석(필수/선택·가능 인원·피하면 좋은 시간)은 결과 화면이 담당합니다.

자세한 설명: [docs/case-study.md](docs/case-study.md) · 시나리오 정의: [docs/cases.md](docs/cases.md)

## 실행 방법

```bash
# 1) 의존성 설치
npm install

# 2) Supabase로 실제 저장하려면 환경변수 설정 (.env.example 참고)
cp .env.example .env.local
#   NEXT_PUBLIC_SUPABASE_URL, SUPABASE_SECRET_KEY 채우기
#
#   로컬 개발 모드에서는 두 값이 비어 있어도
#   .modu-local-db.json 파일 저장소로 회의 생성 플로우를 테스트할 수 있음
#   Vercel 배포 환경에서는 두 값이 비어 있으면 데모 링크 생성 흐름으로 동작

# 3) Supabase를 사용할 경우 테이블 생성 (아래 "Supabase 설정" 참고)

# 4) 개발 서버
npm run dev        # http://localhost:3000

# 그 외
npm run typecheck  # 타입 체크
npm run test       # 후보 계산 알고리즘 단위 테스트
npm run build      # 프로덕션 빌드
```

## 필요한 환경변수

| 변수 | 필수 | 설명 |
| --- | --- | --- |
| `NEXT_PUBLIC_SUPABASE_URL` | 실제 저장 시 필수 | Supabase 프로젝트 URL. 개발 모드에서는 비워두면 파일 저장소를 사용 |
| `SUPABASE_SECRET_KEY` | 실제 저장 시 필수 | 서버 전용 secret key. **브라우저에 노출 금지** |
| `SUPABASE_SERVICE_ROLE_KEY` | 선택 | `SUPABASE_SECRET_KEY` 가 없을 때의 fallback |
| `NEXT_PUBLIC_APP_URL` | 선택 | 안내용. 공유 링크는 브라우저 origin 으로 생성됨 |

> Supabase 접근은 **서버에서만** 이뤄지며, secret/service-role key 는 클라이언트 번들에 포함되지 않습니다. 개발 모드에서 Supabase 값이 없으면 `.modu-local-db.json` 파일 저장소로 동작합니다. Vercel 배포 환경에서 Supabase 값이 없으면 프론트 흐름 확인용 데모 링크 생성 모드로 동작합니다.

## Supabase 설정

`supabase/schema.sql` 전체를 Supabase SQL Editor 에 붙여 실행하세요. 다음을 포함합니다.

- enum 타입 (`attendance_type`, `response_status`, `availability_status`)
- 테이블: `meetings`, `participants`, `availability_blocks`, `meeting_votes`(deprecated — 앱은 더 이상 쓰지 않음, 호환용 유지), `confirmed_slots`
- 제약(날짜·시간 순서), 인덱스
- RLS 활성화 + `anon`/`authenticated` 직접 접근 차단, `service_role` 만 허용

## 후보 계산 알고리즘 동작 방식

후보 계산·해석 로직은 UI 와 분리되어 `lib/scheduler/` 에 있고 단위 테스트로 검증됩니다.

1. **후보 생성** (`generateSlots`): 오늘부터 일정 마감 날짜까지, 주말을 포함한 모든 날의 00:00~24:00 범위에서 30분 단위 후보를 만든다.
2. **제외**: 필수 참석자의 불가능 시간대(`busy`)와 겹치면 후보에서 제거한다.
3. **채점** (`scoreSlot`): 기본 점수(`100`)에서 선택 참석자 `busy`(−15), 필수 참석자 미응답(−12)을 감점한다.
4. **등급**: `가장 추천 / 추천 / 조건부 추천 / 주의 필요`. 필수 전원 가능 여부와 선택 참석자 불가 인원 수로 결정하며, '가장 추천'은 최상위 1개에만.
5. **정렬**: 점수 내림차순, 동점이면 더 이른 시간을 먼저 둔다.
6. **설명** (`explainRecommendation`): 점수 대신 사람이 읽는 한국어 이유를 생성.
7. **맥락형 해석**(`lib/scheduler/contextualResult.ts`): 전체 응답을 해석해 상황(대부분 가능/보통/바쁨/없음)별 해석 문장, 후보 그룹(비슷한 후보는 같은 그룹), 캘린더 신호를 만든다. 현재 캘린더 UI는 `lib/scheduler/calendarDateQuality.ts`가 날짜별 요약을 점수화한 뒤 전체 기간 안의 상대 순위에 따라 원 3개로 추천도를 보여준다. **modu는 회의 시간을 확정하지 않는다** — 최종 시간은 캘린더와 판단 문구를 바탕으로 참여자들이 외부에서 정한다.

미응답자는 후보에서 임의로 가능 처리하지 않고 잠정 결과 문구와 날짜·시간 조회 결과에 `미응답 N명`으로 표시합니다.

> 참여자 응답은 **불가능한 시간(`busy`)만** 받으므로 현재 가점(`preferred`)·비선호(`avoid`) 신호는 입력되지 않습니다. 채점 로직에는 두 상수가 남아 있으나 실제로는 동작하지 않으며, 점심 직후 감점 같은 시간대 가중치는 없습니다.

> 시간은 모두 **Asia/Seoul(KST, +09:00)** 벽시계 기준으로 계산하고, DB 에는 절대 시각으로 저장합니다.

## 현재 구현된 기능

- 일정 생성: 한 항목씩 입력하면 상단 안내 문장이 완성되는 "문장 빌더" 흐름(일정 이름 → 일정 내용 → 장소 → 예상 소요 시간 → 일정 후보 마지막 날 → 응답 마감일 순). 참여자는 생성 단계에서 미리 입력하지 않으며, 생성된 공용 링크로 각자 참여한다.
- 일정 확인: 6개 입력을 모두 채우고 마지막 단계의 `다음`을 누르면 생성 전 `일정 확인` 화면을 거친다. 확인 문장이 순차 강조되고, 각 값은 눌러 수정할 수 있다. 마지막 문구가 나온 뒤 하단 `일정 만들기` 버튼으로 실제 생성한다.
- 일정 생성 직후 `일정이 만들어졌어요` 화면(`MeetingCreatedPanel`): 완료 애니메이션과 일정 요약, 공용 참여 링크를 보여준다. 아직 참여자가 없으면 링크를 받은 사람이 이름이나 별명으로 직접 참여한다는 안내를 표시하고, 하단에서 `링크 복사`와 `내 시간 입력하기`를 제공한다.
- 참여자 응답은 링크 진입 시 일정 안내 → 이름이나 별명 입력 후 자동 참여 → **참여하기 어려운 시간 2단계** → 입력 확인 → 제출 순으로 진행한다. 이름·직무 명단 검증은 하지 않으며, 같은 이름은 구분되는 별명을 요청한다. 참여 권한은 브라우저에 저장되어 같은 브라우저에서 응답을 이어서 수정할 수 있다.
- 제출 후 흐름: 입력 확인(review) → **대기(waiting)** → **캘린더(done)** 이동. 대기 화면은 세로 스텝퍼 + 카운트다운(데모용 7초, 실제 서비스에서는 응답 마감 시각까지의 남은 시간으로 대체 예정)이며, 사람 아이콘과 함께 `{응답}/{전체} 응답 완료`·`응답 마감까지 mm:ss` 응답 현황을 보여준다. 마감(카운트다운 0) 전이라도 **전원 응답이 완료되면** 캘린더를 볼 수 있고(둘 중 먼저), 열람 가능해지면 마지막 스텝(`캘린더를 확인해보세요`)이 활성화된다. 캘린더 화면은 현재 `docs/cases.md` 의 평가용 시나리오(`data/demoCases.ts`)로 동작하며, 상황 버튼으로 상황·솔루션·경고 배너를 전환한다(배경: `docs/case-study.md`). 긴 회의 기간에서는 같은 케이스의 불가 패턴을 기간 전체에 분산해 8월·9월 같은 먼 날짜도 자연스럽게 섞여 보이게 한다. 실제 회의 생성/응답과는 아직 연결하지 않았다
- 추천 시간 화면: 혹시 모를 재사용을 위해 `ResultScreen` 코드는 보존하지만 현재 플로우에서는 비활성화했다.
- 캘린더 화면: 월간 달력 셀에 가능/불가 숫자 대신 전체 기간 안의 상대 추천도를 원 3개로 표시한다(상위권 3개, 중간권 2개, 하위권 1개). 날짜를 누르면 `가장 나은 시간` 기준의 날짜 전체 요약과 예외 시간대를 보여주며, 날짜·시간 검색으로 특정 시각의 참석 가능 여부를 바로 확인할 수 있다. 주말도 평일과 동일하게 선택할 수 있고 일정 마감일은 날짜 오른쪽에 표시한다.
- 변경한 블록만 저장(`available` 은 미저장)
- 순수 후보 계산 엔진은 후보 + 한국어 이유 + 참여자별 영향을 제공한다. 기존 필수/선택 데이터도 호환하며, 신규 참여자는 모두 같은 조건으로 계산한다. 현재 결과 UI는 캘린더 우선 흐름이라 후보 리스트 화면은 비활성 보존 상태다.
- 전원 응답 또는 대기 조건 충족 시 맥락형 해석 문장 + 캘린더 신호 + 날짜·시간 조회로 판단 근거를 정리해 보여줌(최종 결정은 참여자들이 외부에서)
- 확정 화면·공유 문구·`.ics` 다운로드는 legacy 호환용으로만 유지(신규 플로우에서는 confirmed_slots 를 만들지 않음)
- 토큰 기반 권한, 서버 전용 Supabase 접근, RLS
- 개발 환경 Supabase 미설정 시 `.modu-local-db.json` 파일 저장소 fallback
- Vercel 배포 환경 Supabase 미설정 시 프론트 흐름 확인용 데모 링크 생성

## 개인정보 보호

- 상세 일정명은 입력·저장하지 않습니다.
- 불가능한 시간(`busy`)만 저장합니다.
- 메모는 선택 입력이며 결과 요약에 원문을 직접 노출하지 않습니다.
- 회의 데이터에 `expires_at`(기본 생성 후 30일)을 두지만, 만료·삭제 안내 문구는 화면에 노출하지 않습니다.

## 아직 제외한 기능 (알파 범위 밖)

Google Calendar 실시간 연동, 로그인/회원가입, 이메일 자동 발송, Slack 연동, 결제, 조직 관리, 반복 회의, 자동 만료 삭제 작업, 참석자별 상세 일정명.

## 다음 개선 우선순위

1. 만료(`expires_at`) 데이터 정리 작업(크론/엣지 함수).
2. 응답 메모를 블록별로 세분화 (현재는 참석자당 1개의 메모를 `participants.memo` 에 저장).
3. 실제 응답 데이터를 캘린더 결과 화면에 연결(`loadCalendarSnapshot → evaluateAllSlots → buildContextualScheduleResult`).
4. 후보 점수 가중치 설정 UI / 공휴일 표시 옵션.
5. 확정 화면 접근 제어 옵션(현재는 meetingId 를 아는 사람에게 공개).

## 기술 스택

Next.js 16 (App Router) · React 19 · TypeScript · Tailwind CSS · Supabase(Postgres) · Vitest

UI는 Tailwind CSS 기반의 프로젝트 내부 컴포넌트를 사용합니다. 버튼, 입력창, 배지, 카드, 날짜 선택기 등은 `components/ui/` 아래에 있는 자체 컴포넌트로 구성되어 있습니다.

본문 텍스트는 Pretendard(`globals.css` 에서 CDN `@import`, Tailwind `fontFamily.sans` 에도 반영)로 렌더링합니다. 이모지는 `TossFace` 폰트로 렌더링하며(`globals.css` 의 `@font-face` `unicode-range` 로 이모지 코드포인트에만 적용), `public/fonts/TossFaceFontWeb.otf`(COLR/CPAL, 전 브라우저 컬러)를 우선 사용하고 `TossFaceFontMac.ttf` 로 폴백합니다. UI 아이콘은 lucide-react 대신 유니코드 이모지를 `components/ui/Emoji.tsx`(TossFace)로 렌더링합니다.
