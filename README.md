# modu

> 모두의 시간, 모두의 회의

**modu**는 단순 일정 투표가 아니라, 필수/선택 참석자가 **불가능한 날짜·시간**만 입력하면 그걸 종합해 **왜 이 시간이 좋은지 설명하는 회의 시간 의사결정 도구**입니다. 로그인 없이 링크로 응답을 받고, 필수 참석자가 추천 후보에 투표한 뒤 다수결 1위 시간으로 회의를 확정합니다. 회의를 만든 사람도 참석자 중 한 명이며, 별도 관리자 개념이나 관리자 페이지는 두지 않습니다.

이 저장소는 실제로 동료에게 링크를 보내 사용할 수 있는 **알파 버전**입니다.

---

## 실행 방법

```bash
# 1) 의존성 설치
npm install

# 2) Supabase로 실제 저장하려면 환경변수 설정 (.env.example 참고)
cp .env.example .env.local
#   NEXT_PUBLIC_SUPABASE_URL, SUPABASE_SECRET_KEY 채우기
#
#   로컬 개발 모드에서는 두 값이 비어 있어도
#   .modu-local-db.json 파일 저장소로 회의 생성 플로우를 테스트할 수 있음
#   Vercel 배포 환경에서는 두 값이 비어 있으면 데모 링크 생성 흐름으로 동작

# 3) Supabase를 사용할 경우 테이블 생성 (아래 "Supabase 설정" 참고)

# 4) 개발 서버
npm run dev        # http://localhost:3000

# 그 외
npm run typecheck  # 타입 체크
npm run test       # 추천 알고리즘 단위 테스트
npm run build      # 프로덕션 빌드
```

## 필요한 환경변수

| 변수 | 필수 | 설명 |
| --- | --- | --- |
| `NEXT_PUBLIC_SUPABASE_URL` | 실제 저장 시 필수 | Supabase 프로젝트 URL. 개발 모드에서는 비워두면 파일 저장소를 사용 |
| `SUPABASE_SECRET_KEY` | 실제 저장 시 필수 | 서버 전용 secret key. **브라우저에 노출 금지** |
| `SUPABASE_SERVICE_ROLE_KEY` | 선택 | `SUPABASE_SECRET_KEY` 가 없을 때의 fallback |
| `NEXT_PUBLIC_APP_URL` | 선택 | 안내용. 공유 링크는 브라우저 origin 으로 생성됨 |

> Supabase 접근은 **서버에서만** 이뤄지며, secret/service-role key 는 클라이언트 번들에 포함되지 않습니다. 개발 모드에서 Supabase 값이 없으면 `.modu-local-db.json` 파일 저장소로 동작합니다. Vercel 배포 환경에서 Supabase 값이 없으면 프론트 흐름 확인용 데모 링크 생성 모드로 동작합니다.

## Supabase 설정

`supabase/schema.sql` 전체를 Supabase SQL Editor 에 붙여 실행하세요. 다음을 포함합니다.

- enum 타입 (`attendance_type`, `response_status`, `availability_status`)

## 추천 알고리즘 동작 방식

추천 로직은 UI 와 분리되어 `lib/scheduler/` 에 있고 단위 테스트로 검증됩니다.

1. **후보 생성** (`generateSlots`): 오늘부터 회의 마감 날짜까지, 근무 시간 안에서 30분 단위 후보를 만든다.
2. **제외**: 필수 참석자의 불가능 시간대(`busy`)와 겹치면 후보에서 제거한다.
3. **채점** (`scoreSlot`): 기본 점수(`100`)에서 선택 참석자 `busy`(−15), 필수 참석자 미응답(−12)을 감점한다.
4. **등급**: `가장 추천 / 추천 / 조건부 추천 / 주의 필요`. 필수 전원 가능 여부와 선택 참석자 불가 인원 수로 결정하며, '가장 추천'은 최상위 1개에만.
5. **정렬**: 점수 내림차순, 동점이면 더 이른 시간을 먼저 둔다.
6. **설명** (`explainRecommendation`): 점수 대신 사람이 읽는 한국어 이유를 생성.
7. **투표**: 모든 참석자가 응답하면 후보 시간대 투표가 열리고, 필수 참석자의 투표가 모두 모였을 때 최다 득표 후보를 회의 시간으로 확정. 동률이면 사람이 고르지 않고 modu 추천순이 더 높은 후보를 우선.

미응답자는 후보를 제외하지 않고 카드에 `미응답 N명`으로 표시합니다.

> 참석자 응답은 **불가능한 시간(`busy`)만** 받으므로 현재 가점(`preferred`)·비선호(`avoid`) 신호는 입력되지 않습니다. 채점 로직에는 두 상수가 남아 있으나 실제로는 동작하지 않으며, 점심 직후 감점 같은 시간대 가중치는 없습니다.

> 시간은 모두 **Asia/Seoul(KST, +09:00)** 벽시계 기준으로 계산하고, DB 에는 절대 시각으로 저장합니다.

## 현재 구현된 기능

- 회의 생성: 한 항목씩 입력하면 상단 안내 문장이 완성되는 "문장 빌더" 흐름(회의명 → 안건 → 장소 → 예상 회의 진행 시간(시간/분) → 회의 마감 날짜 → 응답 마감일 → 참석자 순). 모든 항목 필수이며, 상단 문장의 값을 클릭하면 해당 항목으로 돌아가 수정. 회의명은 최대 20글자, 안건은 최대 30글자, 장소는 최대 20글자까지 허용하고 초과 시 입력 라벨이 아이콘 포함 경고 문구로 바뀐다. 회의 마감 날짜는 요일을 표시하는 중앙 모달형 커스텀 캘린더(`components/ui/DatePicker.tsx`, KST 안전, 최소 오늘+2일)로 선택하고, 응답 마감일은 회의 마감 날짜 2일 전까지만 고를 수 있다(그 이후 날짜는 달력에서 비활성화되고 호버 시 사유 툴팁, 범위 초과 선택·회의 마감일 당김 시 자동 보정 토스트). 참석자는 가상 직원 50명 검색·클릭으로 추가하고 필수참석/선택참석 두 구역에서 드래그 앤 드롭으로 분류(데스크톱 마퀴 선택, 모바일 터치 드래그 지원)하며 2~8명까지 선택 가능하다. 생성 시 참석자별 토큰 발급, 참석자 링크 생성
- 회의 확인: 폼을 모두 채우고 마지막 단계의 `다음`을 누르면 생성 전 `회의 확인` 화면을 거친다. 1~7번 문장절이 아래→위 페이드인(`fade-up-blur`)으로 **천천히 순차 등장**하고, 각 값은 클릭하면 해당 항목으로 돌아가 수정할 수 있는 파란 키워드다(`수정하려면 키워드를 눌러 해당화면으로 이동하세요` 안내). 마지막 7번 문구가 나온 뒤에야 하단 `회의 만들기` 버튼이 페이드인되며, 이 버튼으로 실제 생성한다
- 회의 생성 직후 `회의가 만들어졌어요` 화면(`MeetingCreatedPanel`): 완료 애니메이션(체크는 즉시) + 제목·회의 요약 카드·참석자 명단·참석자 전달 링크 카드가 위에서부터 순차로 떠오르는 AOS(`fade-up-blur`) + 하단에 균등 2버튼 CTA(`링크 복사` · `회의시간 입력하기`). `회의시간 입력하기`는 새 탭으로 열리고, 하단 CTA는 지연 없이 바로 노출
- 참석자 응답은 문장 빌더 흐름: 링크 진입 시 회의 안내(intro) → `시간 정하러 가기` → 본인 확인(이름 입력(라벨 옆 `?` 아이콘으로 본인확인 이유 툴팁) → 직무 선택(`Select` `variant="menu"` 커스텀 드롭다운), 명단 검증, 오류는 토스트, 성공 시 확인 완료 토스트) → **불가 중심 가능 시간 2단계**(① 불가능한 날짜 `busy` ② 특정 날짜+시간 불가 `busy`, 둘 다 다중 입력, 월 이동(`<`/`>`) 달력·토일 비활성) → 입력 확인(review, `응답 시간 마감 전까지 수정할 수 있어요. 수정하려면 키워드를 눌러 응답화면으로 이동하세요` 안내, 필드값이 클릭 가능한 빨강 키워드) → 제출. 날짜·시간 선택은 공용 `components/scheduler/CalendarModal.tsx`(회의 만들기 `DatePicker` 와 시각적으로 일치)로 뜨며 **모바일은 전체화면 시트, 데스크톱은 가운데 카드**다(모바일 ②는 한 시트에서 날짜+시간을 함께 고르고, 데스크톱은 날짜 모달 후 페이지에서 시간 입력). 근무시간 밖 시간은 컨펌 후 허용, 점심시간과 겹치는 블록은 제외. 같은 브라우저 수정(localStorage 토큰)이며, 복귀 사용자는 대기 화면(waiting)으로 진입
- 제출 후 흐름: 입력 확인(review) → **대기(waiting)** → **추천안 화면(result)** → `캘린더 보기` 로 **캘린더(done)** 이동. 대기 화면은 세로 스텝퍼 + 카운트다운(데모용 7초, 실제 서비스에서는 응답 마감 시각까지의 남은 시간으로 대체 예정)이며, 사람 아이콘과 함께 `{응답}/{전체} 응답 완료`·`응답 마감까지 mm:ss` 응답 현황을 보여준다. 마감(카운트다운 0) 전이라도 **전원 응답이 완료되면** 추천 시간을 볼 수 있고(둘 중 먼저), 열람 가능해지면 마지막 스텝(`추천시간을 확인해보세요`)이 활성화된다. 추천안·캘린더 화면은 현재 `docs/cases.md` 의 8개 케이스를 보여주는 데모(`data/demoCases.ts`)로 동작하며, 상단의 케이스 선택 버튼(1~8, 호버 시 펼침)으로 상황·솔루션·경고 배너를 전환한다. 실제 회의 생성/응답과는 무관
- 추천안 화면: 순위·시간·등급 배지·득표·이유가 담긴 카드 목록(테두리 없이 밀착 드롭섀도우, 호버 시 색 강조). 페이지는 정상 스크롤되고 하단 `캘린더 보기` CTA만 `MobileStickyAction`으로 고정한다(응답 수정은 제공하지 않음 — 수정은 대기 화면에서). 카드를 클릭하면 선택(브랜드 링)되고, 선택 시 하단에서 `투표하기` 버튼이 올라와 `캘린더 보기` 와 세로로 교차한다. 투표하면 `투표됨` 배지가 한 번에 한 추천안에만 붙고 선택은 해제된다. `buildCaseCandidates` 가 §8.5 우선순위(필수 전원 가능 → 필수 많이 → 선택 많이 → 충돌 적음 → 이른 시간)로 정렬하고 등급을 도출
- 캘린더 화면: `buildCaseSnapshot` 이 같은 케이스의 불가(`busy`) 블록을 시간축 그리드에 채워 참석자별 불가능 시간을 보여줌
- 변경한 블록만 저장(`available` 은 미저장)
- 추천 후보 + 한국어 이유 + 참석자별 영향 + 필수/선택 즉시 재계산
- 전원 응답 후 후보 시간대 투표, 필수 참석자 투표 후 최다 득표 후보 확정
- 확정 후 공유 문구 + `.ics` 다운로드
- 토큰 기반 권한, 서버 전용 Supabase 접근, RLS
- 개발 환경 Supabase 미설정 시 `.modu-local-db.json` 파일 저장소 fallback
- Vercel 배포 환경 Supabase 미설정 시 프론트 흐름 확인용 데모 링크 생성

## 개인정보 보호

- 상세 일정명은 입력·저장하지 않습니다.
- 불가능한 시간(`busy`)만 저장합니다.
- 메모는 선택 입력이며 추천 요약에 원문을 직접 노출하지 않습니다.
- 회의 데이터에 `expires_at`(기본 생성 후 30일)을 두지만, 만료·삭제 안내 문구는 화면에 노출하지 않습니다.

## 아직 제외한 기능 (알파 범위 밖)

Google Calendar 실시간 연동, 로그인/회원가입, 이메일 자동 발송, Slack 연동, 결제, 조직 관리, 반복 회의, 자동 만료 삭제 작업, 참석자별 상세 일정명.

## 다음 개선 우선순위

1. 만료(`expires_at`) 데이터 정리 작업(크론/엣지 함수).
2. 응답 메모를 블록별로 세분화 (현재는 참석자당 1개의 메모를 `participants.memo` 에 저장).
3. 추천 후보 화면에서 날짜별 최적 시간대 설명 강화.
4. 추천 점수 가중치 설정 UI / 주말·공휴일 제외 옵션.
5. 확정 화면 접근 제어 옵션(현재는 meetingId 를 아는 사람에게 공개).

## 기술 스택

Next.js 14 (App Router) · TypeScript · Tailwind CSS · Supabase(Postgres) · Vitest

UI는 Tailwind CSS 기반의 프로젝트 내부 컴포넌트를 사용합니다. 버튼, 입력창, 배지, 카드, 날짜 선택기 등은 `components/ui/` 아래에 있는 자체 컴포넌트로 구성되어 있습니다.

본문 텍스트는 Pretendard(`globals.css` 에서 CDN `@import`, Tailwind `fontFamily.sans` 에도 반영)로 렌더링합니다. 이모지는 `TossFace` 폰트로 렌더링하며(`globals.css` 의 `@font-face` `unicode-range` 로 이모지 코드포인트에만 적용), `public/fonts/TossFaceFontWeb.otf`(COLR/CPAL, 전 브라우저 컬러)를 우선 사용하고 `TossFaceFontMac.ttf` 로 폴백합니다. UI 아이콘은 lucide-react 대신 유니코드 이모지를 `components/ui/Emoji.tsx`(TossFace)로 렌더링합니다.
