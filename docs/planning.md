# modu 제품 기획서

## 1. 서비스 개요

**modu**는 여러 사람이 회의 일정을 잡을 때, 단순히 가능한 시간을 모으는 것을 넘어 **필수 참석자, 선택 참석자, 개인별 선호 날짜와 불가능 날짜, 날짜별 시간대 조정**을 함께 고려해 모두가 납득할 수 있는 회의 시간을 찾도록 돕는 웹 서비스다.

이번 버전은 클릭 데모가 아니라, 실제로 동료 6명에게 링크를 보내 각자 가능한 시간을 입력하고, 모든 참석자가 후보 시간대에 투표한 뒤 다수결로 회의 시간을 확정할 수 있는 **실사용 가능한 알파 버전**을 목표로 한다.

핵심 카피:

> 6명의 조건을 비교해, 모두가 납득할 수 있는 1시간을 찾아요.

modu는 단순 일정 투표 서비스가 아니라, "왜 이 시간이 좋은지", "누구에게 어떤 부담이 있는지", "필수 참석자는 모두 가능한지"를 설명하는 **회의 시간 의사결정 도구**다.

## 2. 핵심 문제 정의

여러 사람이 회의 일정을 잡을 때 사용자는 단순히 빈 시간을 찾는 것이 아니다.

실제로는 다음 조건들이 함께 고려된다.

- 반드시 참석해야 하는 사람
- 참석하면 좋지만 필수는 아닌 사람
- 개인별 선호 날짜와 시간대
- 개인별 불가능 날짜와 시간대
- 아직 응답하지 않은 사람으로 인한 불확실성

기존 캘린더나 일정 투표 서비스는 이런 맥락을 충분히 설명하지 못한다. 그 결과 주최자는 메신저 대화, 기억, 눈치, 수작업 비교에 의존해 판단 부담을 떠안는다.

modu는 이 부담을 줄이기 위해 후보 시간을 단순 나열하지 않고, 각 후보가 왜 추천되는지 한국어로 설명한다.

## 3. 알파 버전 목표

이번 버전의 목표는 다음이다.

- 주최자가 회의를 만들 수 있다.
- 참석자 6명에게 공유 링크를 보낼 수 있다.
- 참석자는 로그인 없이 자신의 선호 날짜·시간대와 불가능 날짜·시간대를 입력할 수 있다.
- 주최자는 참석자들의 선호/불가능 시간대를 전체 캘린더로 확인할 수 있다.
- 모든 참석자가 응답하면 후보 시간대 투표가 열린다.
- 추천 후보는 최소 3개, 가능하면 5개까지 제공된다.
- 각 추천 후보는 사람이 이해할 수 있는 이유를 함께 보여준다.
- 모든 참석자의 투표가 모이면 주최자가 최다 득표 후보를 회의 시간으로 확정할 수 있다. 동률이면 주최자가 1위 후보 중 하나를 고른다.
- 확정 후 공유 문구와 `.ics` 캘린더 파일을 받을 수 있다.

## 4. 제외 범위

이번 알파 버전에서는 범위 확장을 막기 위해 다음 기능을 제외한다.

- Google Calendar 실시간 연동
- 로그인/회원가입
- 이메일 자동 발송
- Slack 연동
- 결제
- 조직 관리
- 반복 회의
- 자동 만료 데이터 삭제 작업
- 참석자별 상세 일정명 입력

단, 만료일 `expiresAt`은 저장하고, 서비스 화면에서는 "회의 데이터는 만료일 이후 삭제될 수 있음"을 안내한다.

## 5. 주요 사용자 흐름

### 5.1 랜딩 화면

경로: `/`

랜딩 화면은 사용자가 30초 안에 서비스를 이해하도록 구성한다.

필수 요소:

- 서비스명: `modu`
- 메인 카피: `모두의 시간, 모두의 회의`
- 서브 카피: `회의를 생성하여 참여자들에게 링크를 전달하세요.`
- CTA: `회의 만들기`

### 5.2 회의 만들기 화면

경로: `/meetings/new`

주최자가 회의의 기본 조건을 입력한다. 페이지 배경은 `rgb(255 255 255 / 0.95)`(모든 페이지·헤더 공통)이며, 카드 없이 라벨 텍스트와 입력창으로 구성한다.

입력은 한 화면 안에서 한 단계씩 순서대로 받는다. 한 단계를 `다음`으로 완료하면 다음 단계가 노출되고, 그 전에는 보이지 않는다.

이 화면은 "문장 빌더" UX를 따른다. 주최자가 한 항목씩 입력할수록 페이지 상단에 한국어 안내 문장이 한 절씩 완성되고, 하단에는 현재 단계의 입력창과 `다음` 버튼이 위치한다.

- 상단 문장에서 이미 입력한 값을 클릭하면 그 항목 단계로 돌아가 수정할 수 있다.
- 아직 채우지 않았거나 지운 값 자리에는 회색 점 3개가 파도타기(dot-wave)로 움직이는 자리표시를 보여준다.
- 새 절·단계는 fade 애니메이션으로 등장한다.
- 모든 항목은 필수다.

1. 회의명
2. 안건
3. 장소
4. 회의 마감 날짜
5. 예상 회의 진행 시간(시간, 분 직접 입력)
6. 참석자(모달에서 가상 직원 검색/선택, 최소 2명, 최대 8명)

입력 제한:

- 회의명은 공백 포함 최대 20글자까지 입력할 수 있다.
- 회의 안건은 공백 포함 최대 30글자까지 입력할 수 있다.
- 회의 장소는 공백 포함 최대 20글자까지 입력할 수 있다.
- 제한을 넘기면 별도 문구를 추가하지 않고 기존 입력 라벨이 `!` 아이콘이 포함된 빨간 경고 문구로 바뀌며, 입력창 테두리도 같은 빨간색으로 바뀐다.
- 입력창 포커스 테두리는 `#4593fc` 단일 색상으로 통일하고, 포커스 시 테두리 두께만 1px 더 두껍게 한다.

최종 안내 문장 형태:

> 이번 회의명은 {회의명}에요. 회의 안건은 {안건} 입니다. 회의 장소는 {장소}이며, {날짜(요일 포함, 예: 2026년 7월 5일 일요일)}까지는 회의가 완료되어야 해요. 예상 회의 진행 시간은 {시간}시간 {분} 분입니다.
>
> 회의 참석자 명단은 {이름1}, {이름2}, … 총 N명 입니다.

시간 값이 0이면 시간 부분을, 분 값이 0이면 분 부분을 문장에서 생략한다.

기본값:

- 예상 회의 진행 시간: 1시간 0분
- 회의 마감 날짜: 오늘 기준 7일 뒤
- 내부 계산용 근무 시간: 09:00~18:00
- 날짜 범위: 오늘부터 회의 마감 날짜까지
- 참석자: 비어 있는 상태로 시작하며, 최소 2명 이상 최대 8명까지 직접 추가한다.

회의 마감 날짜는 디자인 톤에 맞춘 커스텀 캘린더(`components/ui/DatePicker.tsx`)로 고른다. 요일을 함께 표시하고, KST 기준으로 안전하게 계산하며, 최소 선택일은 오늘로 제한한다.

Enter 키는 현재 단계를 `다음`으로 진행만 하고 폼을 제출하지 않는다. 실제 제출은 `회의 만들기` 버튼으로만 한다. `회의 만들기` 버튼은 참석자 선택까지 완료(2명 이상 + 모달 닫음)했을 때만 노출한다.

주최자는 근무 시작/종료 시간이나 점심 시간을 직접 입력하지 않는다. 이번 알파에서는 모든 회의가 위 기본 시간 기준으로 계산된다.

참석자 선택 방식:

- 50명의 가상 직원 데이터에서 이름 또는 직책으로 검색한다.
- 가상 직원은 이름과 직책을 가진다.
- 모달은 위/아래 2단 구성이다. 위에는 검색창(왼쪽)과 가상 직원 50명 목록, 아래에는 `필수참석`·`선택참석` 두 구역이 있다.
- 직원 목록 항목을 클릭하면 참석자로 추가·해제(토글)된다.
- 참석자는 최대 8명까지 선택할 수 있으며, 8명이 되면 아직 선택하지 않은 직원은 추가할 수 없다.
- 선택된 참석자는 둥근 뱃지(pill) 형태로 아래 구역에 표시된다.
- 새로 추가한 참석자는 기본적으로 `필수참석` 구역으로 들어간다(기본 `attendanceType=required`).
- 필수↔선택 이동은 뱃지를 드래그 앤 드롭으로 옮긴다. 데스크톱에서는 빈 공간을 드래그해 사각형(마퀴)으로 여러 명을 한 번에 선택해 함께 옮길 수 있고, 모바일에서는 참석자 pill을 터치 드래그해 필수참석↔선택참석 구역으로 옮길 수 있다.
- 선택 인원 뱃지는 데스크톱에서만 작게 표시하고, 모바일에서는 공간 절약을 위해 숨긴다.

참석자는 기본 화면 안에 길게 펼치지 않고 모달에서 검색/선택한다. 주최자는 이 단계에서 필수 참석자를 분류한다.
참석자 단계는 `참석자 선택하기` 버튼으로 모달을 열고, 선택한 참석자는 상단 안내 문장에 이름과 총 인원으로 반영된다.

모달 동작:

- 모바일에서는 모달을 전체 화면으로 띄우고, 데스크톱에서는 고정 크기에 내부 스크롤을 사용해 내용이 늘어도 크기가 유지된다. 직원 목록 영역은 위/아래에 페이드를 둔다.
- 바깥(backdrop) 클릭이나 Esc로 닫을 수 있다.
- 열릴 때 모달 안으로 포커스를 옮기고, 닫힐 때 트리거 버튼으로 복원하며, 모달 안에서는 focus trap을 적용한다.
- 참석자가 2명 미만일 때 `선택 완료`를 누르면 모달을 닫지 않고 "참석자는 최소 2명 이상 선택해 주세요" 유효성 경고를 모달 안에 노출한다.
- `선택 완료`는 전체폭 버튼으로 표시한다.

모바일 하단 CTA:

- 모바일에서 하단 입력 영역과 CTA는 스티키 형태로 같은 위치를 유지한다.
- 하단 영역 자체는 불투명한 본문 배경색을 유지하고, 본문과 하단 영역 사이 맞닿는 부분만 그라데이션으로 부드럽게 이어지게 한다.
- 하단 영역을 테두리, 그림자, 블러, 다른 배경색으로 별도 바처럼 보이게 만들지 않는다.

### 5.3 회의 생성 후 공유 화면

경로: `/meetings/[meetingId]/share`

회의를 만들면 다음 값을 생성한다.

- 고유한 `meetingId`
- 참석자별 `participantToken`

화면에는 참석자용 공유 링크를 보여준다.

- 참석자용 공유 링크: `/m/[meetingId]`

필수 안내 문구:

- 참석자 링크는 누구나 응답할 수 있는 링크다.

### 5.4 참석자 응답 화면

경로: `/m/[meetingId]`

참석자는 로그인 없이 공유 링크로 접속한다.

참석자 본인 확인 방식:

- 참석자는 이름을 입력한다.
- 참석자는 직무를 셀렉트박스에서 선택한다.
- 서버는 입력한 이름과 직무가 실제 참석자 명단에 있는지 검증한다.
- 검증을 통과한 참석자만 날짜 입력 화면으로 이동한다.
- 명단에 없는 사람은 응답할 수 없다.

화면 진입:

- 링크 진입 시 먼저 회의 안내(intro) 화면에서 회의 내용을 문장으로 보여주고, `시간 정하러 가기`로 본인 확인으로 이동한다.
- 본인 확인은 이름 → 직무를 한 단계씩 입력하는 문장 빌더이며, 검증 오류는 토스트로 알리고 성공 시 `확인 완료` 토스트를 띄운다.

가능 시간 입력 방식(문장 빌더, 5단계):

- 회의 만들기와 같은 톤으로, 하단 질문에 답하면 상단에 한국어 문장이 쌓인다. 각 절은 클릭해 수정할 수 있고, 단계는 비워도 넘어갈 수 있다.
  1. 공통적으로 피하고 싶은 시간대(예: 점심) — 시작~종료 시간 여러 개. → `avoid`(모든 날짜)
  2. 공통 선호 시간대 — 여러 개. → `preferred`(모든 날짜)
  3. 불가능한 날짜 — 오늘~마감일 사이 유효 날짜만 활성화된 월 달력에서 다중 선택. → `busy`(하루)
  4. 선호하는 날짜 — 달력 다중 선택. → `preferred`(하루)
  5. 특정 날짜+시간 불가 — 달력에서 날짜를 고르고 그 날 안 되는 시간대 여러 개. → `busy`(날짜+시간)
- 입력하지 않은 시간은 기본 "가능"으로 본다.

달력·시간 입력 동작:

- 달력은 한 번에 한 달만 보여주고 `<` `>` 버튼으로 월을 이동한다(유효 날짜가 있는 월 범위 안에서만). 크기는 회의 만들기 `DatePicker`와 비슷하게 작다.
- 토·일요일은 비활성으로 표시해 선택할 수 없다.
- 선호하는 날짜 달력에서, 앞서 불가능으로 고른 날짜는 빨간색 비활성 + `불가` 배지로 표시해 선택할 수 없다.
- 시간대 입력은 근무시간(09:00~18:00) 밖을 막고, 선호 시간대가 앞서 입력한 "피하고 싶은" 시간대와 겹치면 경고와 함께 추가를 막는다.

데이터 저장 정책:

- 입력값을 우선순위(특정 날짜+시간 `busy` > 불가능 날짜 `busy` > 선호 날짜 `preferred` > 공통 피하고 싶은 `avoid` > 공통 선호 `preferred`)로 30분 셀로 환산해 블록으로 저장한다.
- "가능" 상태는 저장하지 않는다. 점심 시간이 설정된 회의에서는 점심과 겹치는 셀을 제외한다.
- 미응답자는 추천 결과에서 "미응답자 있음"으로 표시한다.

응답 제출 후:

- 같은 브라우저에서는 기존 응답을 불러와 수정할 수 있다.
- 참석자는 단순 완료 화면 대신 일주일 그리드 형태의 회의 캘린더 화면으로 이동한다.
- 회의 캘린더는 날짜 컬럼 × 30분 시간 행 그리드이며, 화면을 넓게(대시보드) 쓰고 마우스 드래그와 휠로 가로 스크롤할 수 있다. 시간(KST) 축은 좌측에 고정된다.
- 각 칸은 `선호`(초록)·`불가능`(빨강) 그룹으로 참석자 이름을 직접 보여주며, 이름 색은 참석자별로 다르다. 가능·미응답은 표시하지 않는다.
- 모든 시간 칸은 동일한 높이를 유지한다.
- 캘린더 화면에서는 응답 수정 액션을 제공한다.
- 같은 브라우저로 다시 접속한 제출 완료 참석자도 완료 카드가 아니라 캘린더 화면을 먼저 본다.
- 모든 참석자가 응답하기 전에는 후보 투표 대신 대기 안내를 보여준다.
- 모든 참석자가 응답하면 후보 시간대 목록이 열리고, 참석자는 후보 중 하나에 투표할 수 있다.

수정 정책:

- 제출 후 같은 브라우저에서는 수정할 수 있어야 한다.
- 이를 위해 첫 제출 시 `participantToken`을 브라우저 localStorage에 저장한다.
- 이후 수정 요청은 `participantId`와 `participantToken`이 일치할 때만 허용한다.

개인정보 정책:

- 상세 일정명은 입력받지 않는다.
- 메모는 선택 입력이다.
- 메모는 기본적으로 추천 요약에 원문 그대로 노출하지 않는다.
- 주최자에게는 상세 사유보다 충돌 요약을 중심으로 보여준다.

### 5.5 주최자 결과 화면

경로: `/admin/[meetingId]/[adminToken]`

주최자는 admin token이 포함된 주최자 화면에서 결과를 확인한다. 공유 화면에서는 참석자 링크만 노출한다.

화면에 보여줄 정보:

- 회의명
- 회의 조건 요약
- 참석자별 응답 완료 여부
- 미응답자 목록
- 참석자들이 입력한 선호/불가능 시간을 집계한 전체 캘린더
- 현재 응답 기준 추천 후보
- 후보별 투표 수
- 다수결 확정 상태

추천 후보:

- 최소 3개
- 가능하면 5개까지 표시
- 각 후보 카드에는 다음 정보를 포함한다.
  - 시간
  - 추천 등급
  - 필수 참석자 충족 여부
  - 선택 참석자 참석 가능 수
  - 비선호 충돌 수
  - 추천 이유
  - 참석자별 영향 요약

주최자가 필수/선택 참석 여부를 변경하면 추천 결과가 즉시 다시 계산되어야 한다.

확정 정책:

- 참석자 전원이 응답하기 전에는 후보 투표를 시작하지 않는다.
- 참석자 전원이 후보에 투표하기 전에는 회의 시간을 확정하지 않는다.
- 투표가 모두 모이면 주최자가 최다 득표 후보를 확정한다. 확정은 주최자만 수행한다.
- 1위가 동률이면 주최자가 동률 후보 중 하나를 골라 확정한다.
- 회의가 확정되면 참석자의 응답과 투표를 잠근다.
- 참석자가 응답을 수정하면 해당 회의의 후보 투표는 초기화된다.

### 5.6 확정 화면

경로: `/meetings/[meetingId]/confirmed`

확정 화면에는 다음 정보를 보여준다.

- 확정된 회의 시간
- 회의 길이
- 참석 대상 요약
- 필수 참석자 충족 여부
- 공유용 문구
- 공유 문구 복사 버튼
- `.ics` 캘린더 파일 다운로드 버튼

실제 이메일 발송이나 캘린더 초대 발송은 하지 않는다.

공유 문구 예시:

```txt
[modu] 회의 시간이 확정되었습니다.
회의명: 주간 제품 회의
시간: 2026년 7월 1일 수요일 14:00~15:00
필수 참석자는 모두 참석 가능한 시간입니다.
```

## 6. 개인정보 보호 정책

제품 안에 반드시 다음 원칙을 반영한다.

- 상세 일정명은 입력받지 않는다.
- 가능 여부와 선호 상태만 저장한다.
- 메모는 선택 입력이며, 기본적으로 추천 요약에 직접 노출하지 않는다.
- 주최자는 참석자별 상세 사유가 아니라 충돌 요약을 중심으로 본다.
- 회의 데이터는 `expiresAt` 이후 삭제될 수 있다는 안내를 표시한다.
- 이 서비스는 민감한 일정 내용을 저장하지 않는다는 마이크로카피를 주요 화면에 넣는다.

## 7. 성공 기준

알파 버전은 다음 조건을 만족하면 성공으로 본다.

- 실제 동료 6명에게 링크를 보내 응답을 받을 수 있다.
- 주최자가 응답 현황과 추천 후보를 이해할 수 있다.
- 주최자가 전체 캘린더에서 선호/불가능 시간 집계를 이해할 수 있다.
- 참석자가 제출 후 바로 일주일 그리드 캘린더에서 전체 응답 집계를 이해할 수 있다.
- 추천 결과가 단순 점수표가 아니라 납득 가능한 설명을 제공한다.
- 필수 참석자와 선택 참석자의 차이가 추천 결과에 반영된다.
- 선호 날짜/시간대와 불가능 날짜/시간대가 결과에 반영된다.
- 참석자 전원 응답 후 후보 투표가 가능하다.
- 참석자 전원 투표 후 최다 득표 후보를 확정할 수 있다. 1위가 동률이면 주최자가 동률 후보 중 하나를 고른다.
- 회의 확정 후 공유 가능한 문구와 캘린더 파일을 제공한다.

## 8. 다음 버전(v2) 방향 — 더 똑똑한 추천과 "모두 안 될 때" 대응

> 이 장은 실사용 알파(1~7장) 이후 modu가 나아갈 제품 방향을 정리한 것이다. 기획 논의에서 **확정한 6가지 결정**과, 그에 따라 바뀌는 동작을 누구나 바로 이해할 수 있게 적는다. (1~7장은 현재 알파의 명세, 8장은 그 위에 얹을 다음 방향이다.)

### 8.1 한 줄 목표

참석자가 가능 시간을 다 입력하고 나면 **고민을 최소화**하고, modu가 **"이 시간이 좋아요"를 이유와 함께 먼저 제시**한다. 특히 **"필수 참석자가 다 되는 날"이 항상 우선**되고, **모두 되는 날이 없을 때도 최선의 차선**을 제시한다.

### 8.2 지금 방식의 한계 (왜 바꾸나)

1. **단계가 많다.** 지금은 `전원 응답 → 전원 투표 → 주최자 확정`. modu가 사실상 답을 계산할 수 있는데도 사람이 다시 줄 세운다.
2. **추천 순위가 직관과 어긋날 수 있다.** 지금은 여러 조건을 점수로 합산해서, 가끔 **"필수는 다 되지만 선호가 적은 시간"** 이 **"필수 1명이 빠지지만 선호가 많은 시간"** 보다 낮게 나온다. (필수 충족이 더 중요한데 점수에서 밀리는 문제)
3. **추천이 0개가 될 수 있다.** 지금은 필수 참석자가 한 명이라도 안 되는 시간은 후보에서 통째로 버린다. 그래서 "모두 되는 날"이 없으면 추천이 아예 안 나오고, 결정 부담이 다시 주최자에게 넘어간다.

### 8.3 확정된 6가지 결정

| # | 무엇을 정했나 | 선택 | 왜 / 효과 |
|---|---|---|---|
| 1 | 응답을 다 모은 뒤 흐름 | **추천을 먼저 보여주되, 확정은 지금처럼 참석자 투표로** | 투표(합의감)는 유지하되, 빈 화면이 아니라 modu가 고른 후보 위에서 투표하게 해 고민을 줄인다. |
| 2 | 모두 되는 날이 없을 때 | **그 시간에 못 오는 '필수 1명'에게 일정 조정 요청을 보내고, 끝내 안 되면 빼고 진행** | "어쩔 수 없는 날"을 modu가 짚어주고, 정중하게 한 번 더 물은 뒤 진행한다. |
| 3 | 참석자 중요도 | **지금처럼 '필수참석 / 선택참석' 2가지만** (사람별 별표·점수 없음) | 입력을 늘리지 않고 단순하게 유지한다. |
| 4 | "누가 못 오는지" 공개 범위 | **참석자 모두에게 이름까지 공개** | 누가 왜 안 되는지 투명하게 보여 납득을 돕는다. |
| 5 | 일정 조정 요청 전달 방법 | **그 사람이 자기 응답 링크에 다시 들어오면 화면에 요청 배너 표시** | 로그인·이메일 없이도 modu 방식(링크 기반)으로 전달한다. (4장 '이메일 자동 발송 제외'와도 일관) |
| 6 | 추천 1등을 고르는 규칙 | **'필수 참석자가 최대한 많이 되는 시간'을 항상 1순위** | 단순·예측 가능. "필수는 되고 선택만 안 되는 날"이 자연히 1등이 된다. |

### 8.4 바뀐 전체 흐름 (사용자 입장)

1. **참석자 전원이 가능 시간 입력 완료** → modu가 후보 시간들을 **'필수가 가장 많이 되는 순서'로 자동 정렬**해, 좋은 후보 3~5개를 이유와 함께 보여준다.
2. **참석자는 그 후보 중에 투표**한다(현재와 동일). 주최자는 최다 득표 후보를 확정한다.
3. **모두 되는 날이 없으면(필수 1명이 꼭 빠지는 상황)** → modu가 **"가장 적게 빠지는 차선은 X요일 (못 오는 사람: OO님)"** 을 보여준다.
   - 주최자는 **OO님에게 '일정 조정 요청'을 보낼 수 있다.**
   - OO님이 자기 응답 링크에 다시 들어오면 **"이 시간으로 조정 가능하세요?" 배너**를 보고 답한다.
   - 조정되면 모두 가능한 시간이 생기고, **끝내 안 되면 OO님을 빼고 그 시간으로 확정**한다.
4. 모든 후보 카드에는 **그 시간에 못 오는 사람의 이름**이 전원에게 보인다.

### 8.5 "추천 1순위 규칙"을 쉽게 (6번 자세히)

modu는 후보를 다음 우선순위로 줄 세운다. **위 기준이 같을 때만** 아래 기준으로 내려간다.

1. **필수 참석자가 다 되는가** — 되는 시간이 항상 먼저
2. **필수 참석자가 더 많이 되는가** — 한 명이라도 더 오는 시간
3. 아직 응답 안 한 필수가 더 적은가 — 불확실성이 작은 시간
4. 선택 참석자가 더 많이 오는가
5. 선호 시간과 더 겹치는가 / 피하고 싶은 시간을 덜 건드리는가
6. 더 이른 시간

**예시 — '화요일' 케이스:**

- 화요일 2시: 필수 4명 **전원 가능**, 선택 1명 불가
- 목요일 10시: 필수 3명만 가능(**1명 빠짐**), 선택 3명 전원 + 선호 많음

→ 지금 방식(점수 합산)이면 선호가 많은 **목요일**이 1등이 될 수 있다. 새 규칙이면 **1번 기준(필수 전원 가능)에서 화요일이 즉시 1등**이고, 선택·선호가 아무리 좋아도 이걸 뒤집지 못한다. "어쩔 수 없이 화요일"이 자동으로 답이 된다.

### 8.6 개인정보 관련 메모

- 4번 결정(불참자 이름 전원 공개)은 **투명성을 우선한 선택**이다. 납득은 쉽지만, 특정인이 "못 오는 사람"으로 드러나는 부담이 있다.
- 추후 부담이 크면 **'주최자에게만 보이기'** 또는 **'이름 없이 인원 수만(예: 필수 4명 중 1명 불가)'** 으로 좁힐 수 있다. (현재 결정은 전원 공개)

### 8.7 현재 알파와 달라지는 점 (요약)

- **그대로 유지**: 참석자 투표로 확정, 필수/선택 2분류, 공유 링크·공유 문구·`.ics` 다운로드.
- **바뀜**: 추천 정렬을 '필수 우선'으로 고정(점수 합산 → 우선순위 규칙), 추천이 0개가 되지 않게 '차선'을 항상 표시, 후보 카드에 불참자 이름 공개.
- **새로 생김**: '일정 조정 요청' 기능 + 그 사람 링크 재방문 시 요청 배너.

### 8.8 단계적 적용 순서 (제안)

1. **1단계 (효과 가장 큼, 화면 변화 작음)**: 추천 정렬을 '필수 우선' 규칙으로 바꾸고, 0개가 안 나오게 '차선'을 보여준다. → "화요일 케이스가 1등 되는지"로 검증.
2. **2단계**: 후보 카드에 불참자 이름 표시 + '일정 조정 요청' 보내기/배너 흐름.
3. **3단계**: 필요 시 추가 정보(예: 회의 긴급도)나 더 정밀한 규칙을 검토한다.
