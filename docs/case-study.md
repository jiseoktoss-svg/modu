# modu 설계 의도 요약 (Case Study)

> 이 문서는 과제 평가자가 "왜 이렇게 만들었는지"를 한 번에 읽을 수 있도록 정리한 설계 의도 요약입니다.
> 상세 기획은 [planning.md](./planning.md), 시나리오 정의는 [cases.md](./cases.md) 참고.

## 1. 문제 정의

여러 명이 참여하는 회의 시간 조율은 다음이 어렵습니다.

- 전원의 빈 시간을 하나하나 맞춰보는 비용이 크다.
- 필수로 와야 하는 사람과 오면 좋은 사람의 무게가 다르다.
- 특정 시간대 회의 불가처럼 "불가능은 아니지만 부담스러운" 제약이 섞여 있다.
- 응답이 다 모이기 전의 결과를 확정처럼 읽으면 사고가 난다.

modu는 이 문제를 **판단 보조** 방식으로 풉니다. 투표도, 자동 확정도 없습니다.
modu는 캘린더와 판단 근거만 제공하고, 최종 회의 시간은 참여자들이 외부(Slack·카톡·구두)에서 정합니다.

## 2. 사용자 가설

- 사람들은 "가능한 시간"을 고르는 것보다 **"어려운 시간"을 짚는 것**을 훨씬 빠르게 한다.
- 결과를 볼 때 궁금한 것은 순위표가 아니라 **"이 시간엔 누가 되고 누가 어렵지?"**라는 구체적 질문이다.
- 색은 순위가 아니라 **신호**로 읽힌다 — 캘린더에서는 피하면 좋은 날만 색으로 드러내면 사용자가 더 덜 오해한다.

## 3. 왜 '피하고 싶은 시간'을 별도로 받지 않았나?

특정 시간대 회의 불가, 집중 업무 시간처럼 성격이 다른 제약을 모두 별도 입력으로 나누면
사용자는 '불가능한 시간'과 '피하고 싶은 시간'의 차이를 직접 판단해야 합니다.

- "불가능이랑 피하고 싶음이 뭐가 다르지?"
- "어느 정도 불편하면 피하고 싶은 시간으로 넣어야 하지?"
- "이걸 입력하면 결과에 얼마나 반영되지?"

이는 입력 단계를 늘리고, 결과 화면에서도 해석해야 할 신호(가능/불가/회피/선호 × 필수/선택 × 미응답)를 과도하게 만듭니다.

따라서 이번 버전에서는 사용자가 **'회의가 어려운 날짜·시간'만** 입력하도록 단순화했습니다.
사용자는 본인이 회의에 참여하기 어렵다고 느끼는 시간을 하나의 방식으로 표시하고,
modu가 그 결과를 필수/선택 참석자와 전체 참석 가능 인원 기준으로 해석합니다.

**입력은 단순하게 유지하고, 판단은 결과 화면에서 설명하는 것**이 이 프로토타입의 핵심 방향입니다.
현재 데모의 "특정 시간대만 어려움" 케이스도 이 하나의 입력으로 흡수됩니다.

## 4. 왜 시나리오 데모인가?

이 과제의 핵심은 여러 사람이 실제로 응답을 완료하는 저장 기능보다,
**복잡한 조건을 사용자가 어떻게 이해하고 판단하는지**입니다.

평가자가 6명의 역할을 나눠 여러 브라우저에서 응답하고 필수참석자 불참·미응답·비슷한 후보 같은
상황을 직접 재현하기에는 시간이 많이 들기 때문에, 결과 화면에는 대표적인 회의 조율 상황을
빠르게 전환해 볼 수 있는 **시나리오 컨트롤**(평가용 시나리오)을 넣었습니다.

각 시나리오는 전원 가능, 선택참석자 일부 불참, 필수참석자 불참, 특정 시간대 불가,
미응답 등 실제 회의 조율에서 판단이 어려운 장면을 압축한 것입니다. 회의 기간이 길면 같은 패턴을 기간 전체에 분산해, 마감일이 먼 캘린더에서도 앞부분 며칠만 피하면 좋은 날로 몰리지 않게 했습니다.

즉, 이 데모는 가짜 데이터가 아니라 **설계 가설을 검증하기 위한 시나리오 기반 평가 장치**입니다.
회의 생성·참석자 응답·저장(Supabase/파일 저장소)은 실제로 동작하며, 결과 화면의 해석 로직
(`lib/scheduler/`)은 실데이터 입력(`SchedulerInput`)을 그대로 받을 수 있게 분리되어 있습니다.

## 5. 핵심 화면별 의도

| 화면 | 의도 |
|---|---|
| 회의 만들기 | 한 항목씩 입력하면 상단 문장이 완성되는 문장 빌더 — 폼이 아니라 대화처럼 읽히게. |
| 참석자 응답 | 본인 확인 → 회의가 어려운 날짜 → 특정 날짜·시간 2단계. 어려운 시간만 짚으면 끝. |
| 회의 캘린더 | 결과 흐름의 중심 화면. 색은 피하면 좋은 날을 알려주는 신호로만 쓴다. 날짜 셀에는 날짜 전체 후보 기준 참석 가능 인원 범위를 보여주고, 날짜를 누르면 **날짜 전체 요약**("회의 가능 시간대 전체에 모든 인원이 참석할 수 있어요" + 예외 시간)을 보여줘 "이 시간만 가능한가?"라는 오해를 막는다. |
| 특정 시간 검색 | "7/15 14시에 누가 되지?"라는 구체적 질문에 바로 답한다. 확정·투표가 아니라 조회 전용. |
| 추천 시간 화면 | 문장형 추천 요약과 후보 그룹 화면 코드는 보존하지만, 현재 플로우에서는 비활성화했다. 필요하면 캘린더 이전 단계로 되살릴 수 있다. |

## 6. 시나리오 설명

1. **전원이 가능한 후보가 여러 개** — 무의미한 1·2·3순위 대신 같은 그룹으로 묶고, 날짜 전체 요약을 검증.
2. **6명 모두 일정이 빽빽함** — 전원이 동시에 비는 시간이 거의 없을 때, 가장 덜 충돌하는 차선 날짜와 반복적으로 피해야 할 시간대를 분리해 보여줌.
3. **필수 vs 선택 충돌** — 선택이 더 많이 가능해도 필수가 어려운 후보는 뒤로.
4. **특정 시간대만 어려움** — 날짜 전체가 아니라 예외 시간대만 피하면 된다는 점을 검증.
5. **필수 1명 어려움** — '필수 1명 어려움' 그룹 분리, 최우선 후보처럼 과장하지 않음.
6. **필수 2명 이상 어려움** — 강한 경고 + 기간 조정 안내.
7. **미응답 잠정 결과** — 미응답자는 가능 인원에서 제외, "필수 모두 가능" 과장 금지.

상세 조건은 [cases.md](./cases.md), 데이터는 `data/demoCases.ts`(동일 구조)에 있습니다.

## 7. 한계와 다음 개선

- 대기 화면의 응답 마감 카운트다운은 데모용 7초 고정 — 실제로는 응답 마감 시각 기반으로 대체.
- 결과 캘린더를 실데이터로 전환: `loadCalendarSnapshot → evaluateAllSlots → buildContextualScheduleResult` 경로는 이미 테스트로 잠가 두었고, 화면 연결만 남음.
- 상대적 빨강 임계값(`RELATIVE_AVOID_GAP_*`, `MAX_RELATIVE_RED_RATIO`)은 시나리오를 돌려보며 조정 여지가 있음.
- 만료 데이터 정리(cron), 블록별 메모, 주말·공휴일 옵션 등은 알파 범위 밖.
