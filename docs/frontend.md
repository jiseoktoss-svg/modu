# modu 프론트엔드 구현 문서

## 1. 프론트엔드 목표

modu의 프론트엔드는 사용자가 30초 안에 서비스 가치를 이해하고, 복잡한 일정 조율 과정을 부담 없이 완료할 수 있도록 구성한다.

핵심 UX 목표:

- 주최자는 빠르게 회의를 만들고 링크를 복사할 수 있다.
- 참석자는 로그인 없이 모바일에서도 쉽게 시간표를 입력할 수 있다.
- 참석자는 모든 응답이 모인 뒤 후보 시간대에 투표할 수 있다.
- 주최자는 전체 캘린더, 추천 후보의 이유, 투표 현황을 한눈에 이해할 수 있다.
- 모든 상태는 색상뿐 아니라 텍스트 라벨로도 표현한다.

## 2. 기술 기준

- Next.js App Router
- TypeScript
- Tailwind CSS
- Server Components 기본 사용
- 상호작용이 필요한 폼, 시간표, 복사 버튼, 추천 재계산 UI는 Client Components로 분리
- 본문 텍스트 폰트는 `Pretendard`(globals.css의 CDN `@import` + `tailwind.config.ts`의 `fontFamily.sans`에 반영)
- 별도 아이콘 라이브러리를 두지 않는다. UI 아이콘은 유니코드 이모지를 `components/ui/Emoji.tsx`(`TossFace` 폰트)로 렌더링한다. (`lucide-react` 의존성 제거)
- 이모지는 `public/fonts/TossFaceFontWeb.otf`(COLR/CPAL, 전 브라우저 컬러)를 우선 사용하고 `public/fonts/TossFaceFontMac.ttf`로 폴백하며, `TossFace` 폰트의 `unicode-range`로 이모지 코드포인트에만 적용된다
- 디자인/컴포넌트는 Tailwind CSS 기반의 프로젝트 내부 컴포넌트를 사용한다. 버튼, 입력창, 카드, 배지, 날짜 선택기, 복사 버튼 등은 `components/ui/` 아래 자체 컴포넌트로 구성한다.
- `next.config.mjs`는 기본 Next.js 설정을 유지한다. 별도 UI 라이브러리 번들 최적화 설정은 두지 않으며, Turbopack(`next dev --turbo`)도 켜지 않는다.
- 토스트 알림은 앱 공통 패턴(상단 고정 pill + 아이콘 + 자동 사라짐)으로 제공한다. (예: 참석자 본인확인 검증 오류, 링크 복사 완료) 모바일에서는 토스트를 화면 폭에 가깝게(가운데 정렬, 텍스트 중앙) 넓혀 어색한 줄바꿈을 줄이고, 문구는 짧게 유지한다.
- 모바일 하단 입력 영역과 CTA는 `MobileStickyAction`을 재사용한다. 상단 내용이 길어져 스크롤이 생겨도 하단 입력/CTA 위치는 유지하며, 하단 영역 자체는 불투명한 본문 배경색을 유지한다. 본문과 맞닿는 부분만 선·그림자 없이 흰색 그라데이션으로 부드럽게 이어지게 한다.
- `MobileStickyAction`은 짧은 화면에서 불필요한 본문 스크롤을 만들지 않는다. 실제 본문 높이가 화면보다 길어질 때만 하단 CTA에 가리지 않도록 필요한 여백을 확보한다.

## 4. 화면별 구현 상세

### 4.1 랜딩 화면

필수 콘텐츠:

- 서비스명: `modu`
- 메인 카피: `모두의 시간, 모두의 회의`
- 서브 카피: `회의를 생성하여 참여자들에게 링크를 전달하세요.`
- CTA: `회의 만들기`

화면 방향:

- 첫 화면에서 제품명과 핵심 카피가 분명히 보여야 한다.
- 랜딩만 읽어도 단순 투표 앱이 아니라 의사결정 도구라는 점이 드러나야 한다.
- 마케팅 페이지처럼 과하게 장식하지 않고, 실제 서비스 진입을 우선한다.
- 모바일 랜딩은 `min-h-dvh` 기준으로 화면 높이를 잡고, 내용이 짧을 때는 본문이 스크롤되지 않아야 한다.

### 4.2 회의 만들기 화면

주요 컴포넌트:

- `MeetingCreateForm`
- `ParticipantListEditor`

화면 톤:

- 페이지 배경은 `rgb(255 255 255 / 0.95)`(Tailwind `bg-white/95`, 모든 페이지·헤더 공통)이며, 카드 없이 라벨 텍스트 + 입력창으로 구성한다.

폼 기본값:

- 회의 길이: 1시간 0분
- 회의 마감 날짜: 오늘 기준 7일 뒤
- 내부 계산용 근무 시간: 09:00~18:00
- 신규 회의의 날짜 범위: 오늘부터 회의 마감 날짜까지
- 참석자: 비어 있는 상태로 시작

문장 빌더 UX(한 페이지 안의 단계형):

- 주최자가 항목을 하나씩 입력할수록 페이지 상단의 한국어 안내 문장이 완성되고, 하단에는 현재 항목 입력창과 `다음` 버튼이 위치한다.
- 상단 문장의 값을 클릭하면 그 항목으로 돌아가 수정할 수 있다.
- 아직 채우지 않았거나 지운 값 자리에는 회색 점 3개의 `파도타기`(dot-wave) 애니메이션을 표시한다.
- 새 절/단계는 fade 애니메이션으로 등장한다.
- 순서: 회의명 → 안건 → 장소 → 회의 마감 날짜 → 예상 회의 진행 시간(시간/분) → 참석자
- 예상 회의 진행 시간 단계에서 `다음`을 누르면 참석자 단계로만 이동한다. 참석자 선택 모달은 참석자 단계의 `참석자 선택하기` 버튼을 눌렀을 때 열린다. 선택 후 상단 문장에 참석자 이름과 총 인원을 표시한다.
- 상단 안내 문장 영역은 좌측 정렬이다(참석자 명단 포함).
- 최종 문장 형태: `이번 회의명은 {회의명}에요. 회의 안건은 {안건} 입니다. 회의 장소는 {장소}이며, {날짜(요일 포함, 예: 2026년 7월 5일 일요일)}까지는 회의가 완료되어야 해요. 예상 회의 진행 시간은 {시간}시간 {분} 분입니다.` 그리고 줄을 바꿔 `회의 참석자 명단은 {이름1}, {이름2}, … 총 N명 입니다.` (시간 값이 0이면 시간 부분, 분 값이 0이면 분 부분을 문장에서 생략한다.)

입력 컨트롤:

- 회의명은 공백 포함 최대 20글자, 회의 안건은 공백 포함 최대 30글자, 회의 장소는 공백 포함 최대 20글자까지 입력할 수 있다.
- 글자 수 제한을 넘으면 기존 입력 라벨이 `!` 아이콘 + 빨간 경고 문구로 바뀌고, 입력창 테두리도 같은 빨간색으로 바뀐다. 별도 경고 문구를 라벨 아래에 추가하지 않는다.
- 입력창 포커스 테두리는 `#4593fc`(Tailwind `brand-400`) 단일 색상으로 통일하고, 포커스 시 테두리 두께만 1px 더 두껍게 한다. 별도 파란 링은 사용하지 않는다.
- 회의 마감 날짜는 디자인 톤에 맞춘 커스텀 캘린더 `DatePicker`(요일 표시, KST 안전, `min=오늘`)로 받는다. `input[type=date]`는 사용하지 않는다.
- `DatePicker` 달력은 화면 중앙 모달로 띄운다. 작은 모바일 화면에서도 모달 자체가 화면 밖으로 넘치지 않게 한다.
- 회의 길이 입력 라벨은 `예상 회의 진행 시간을 입력해주세요`이며, 시간/분은 프로젝트 내부 숫자 스테퍼(+/− 버튼과 직접 입력)로 받는다. 시간/분 입력 영역은 좌측 정렬이다.
- 시간과 분이 모두 0이면 라벨 아래에 `적절한 시간값을 입력해주세요.` 안내를, 9시간 초과 시 근무시간 안내 문구를 표시한다.
- `Enter` 키는 다음 단계로 진행만 하고 폼을 제출하지 않는다. 제출은 `회의 만들기` 버튼으로만 한다.
- `회의 만들기` 제출 버튼은 참석자 선택을 완료(2명 이상 + 모달 닫음)했을 때만 노출한다.

검증(모든 항목 필수):

- 회의명, 안건, 장소, 회의 마감 날짜, 예상 회의 진행 시간이 모두 필수다.
- 회의명은 최대 20글자, 안건은 최대 30글자, 장소는 최대 20글자까지 허용한다.
- 참석자는 최소 2명 이상, 최대 8명까지 허용한다.
- 회의 길이는 시간과 분을 직접 입력하며, 합산 시간이 근무 시간(9시간) 이내여야 한다.

주최자는 근무 시작/종료 시간과 점심 시간을 입력하지 않는다. 화면에서는 회의 마감 날짜만 받고, 서버가 오늘부터 마감 날짜까지의 범위와 09:00~18:00 고정 근무 시간을 저장한다.

참석자 모달:

- 모바일에서는 참석자 선택 모달을 전체 화면으로 띄우고, 내용은 가능한 한 한 화면 안에서 보이도록 컴팩트하게 배치한다. 데스크톱에서는 고정 크기 + 내부 스크롤(내용이 늘어도 크기 유지)을 사용한다.
- 바깥(backdrop) 클릭이나 `Esc`로 닫고, 열릴 때 포커스를 이동하고 focus trap을 둔다.
- 모달 상단에는 `참석자 선택` 타이틀만 두고, 닫기 버튼은 프로젝트 내부 버튼 스타일을 쓴다. 별도 서브타이틀이나 설명 문구는 두지 않는다.
- 레이아웃은 위/아래 구성이다. 위에는 검색 입력창(왼쪽 정렬, 높이·상하 여백을 컴팩트하게 줄임)과 50명의 가상 직원 목록을 두고, 아래에는 `필수참석`·`선택참석` 두 구역을 둔다.
- 멤버 구역 아래에 별도 안내문은 두지 않는다. 대신 `선택 완료`를 참석자 1명 이하인 상태로 누르면 모달을 닫지 않고 `참석자는 최소 2명 이상 선택해 주세요.` 유효성 경고를 모달 안에 띄운다.
- 직원은 이름과 직책을 가지며, 이름 또는 직책으로 검색할 수 있다.
- 직원 목록 항목을 클릭하면 참석자를 추가/해제 토글한다. 참석자가 8명이 되면 아직 선택하지 않은 직원은 추가할 수 없게 비활성화한다.
- 선택된 참석자는 pill(둥근 뱃지) 형태로 아래 구역에 표시된다.
- 새로 추가하면 기본 `필수참석` 구역으로 들어간다(기본 `attendanceType=required`). 필수↔선택 이동은 뱃지 드래그 앤 드롭으로 한다.
- 빈 공간을 드래그하면 사각형(마퀴)으로 여러 명을 한 번에 선택한 뒤 함께 옮길 수 있다. 모바일에서는 선택된 참석자 pill을 터치 드래그해 필수참석↔선택참석 구역으로 옮길 수 있다.
- 직원 목록 영역은 위/아래에 페이드를 둔다.
- 선택 인원 뱃지는 데스크톱에서만 작게 노출하고, 모바일에서는 공간 절약을 위해 숨긴다.
- `선택 완료`는 전체폭 버튼으로 표시한다.

### 4.3 회의 생성 완료 · 공유 화면

주요 컴포넌트:

- `MeetingCreatedPanel`(회의 생성 직후 `회의가 만들어졌어요` 화면)
- `ShareLinksPanel`, `CopyButton`

구성:

- 상단에 완료 애니메이션(체크 + 헤일로 + 버스트, CSS/SVG)과 `회의가 만들어졌어요` 타이틀.
- 회의 내용 요약 카드(회의명·안건·장소·마감일·예상 시간)와 참석자 명단(필수/선택 뱃지).
- `참석자 전달 링크` 카드 + 복사 버튼.
- 하단에는 `수정하기` / `회의시간 입력하기` 2개 버튼을 `MobileStickyAction`(모바일 화면 하단 고정, 본문은 그 위로 스크롤)으로 배치한다.

복사 UX:

- 복사 성공 시 상단에 `복사 완료` 토스트(아이콘 포함)를 잠시 표시한다.
- 링크는 잘리지 않게 한 줄 말줄임으로 보여준다.

반응형:

- 모바일에서는 참석자 명단을 2열 그리드(뱃지 축소), 콘텐츠를 세로 가운데로 모아 링크 카드 위아래 여백을 맞추고 스크롤 없이 한 화면에 담는다. 데스크톱에서는 명단 영역을 더 높게 둔다.

### 4.4 참석자 응답 화면

링크를 받은 참석자는 회의 만들기 화면과 같은 "문장 빌더" 톤으로 응답한다. 하단에서 한 단계씩 입력할 때마다 상단에 한국어 문장이 쌓이고, 상단 문장의 값을 클릭하면 해당 단계로 돌아가 수정할 수 있다. (`ResponseForm`)

흐름:

1. **회의 안내(intro)**: 링크 진입 시 회의 내용을 문장으로 보여주고(`MeetingSummarySentence`, 참석자 이름·명수는 제외), 하단 `시간 정하러 가기` 버튼으로 본인확인 화면으로 이동한다.
2. **본인확인**: 이름 → 직무를 한 단계씩 입력하는 문장 빌더(`저는 ○○이고, 직무는 ○○이에요.`). `확인하고 다음`을 누르면 명단(roster)과 대조해 검증하고, **검증 오류는 아이콘 토스트**로 알린다. 성공하면 다음(가능 시간) 화면으로 넘어간다.
3. **가능 시간(5단계 문장 빌더)**: 하단 질문에 답할 때마다 상단 문장이 이어진다. 단계마다 입력이 없어도 건너뛸 수 있다. 상단 문장에서 **아직 정하지 않은(현재 진행 중) 항목은 회색 점 3개 `파도타기`(dot-wave)** 로 표시하고, 지나온 단계는 답(`…없어요` 포함)으로 확정 표시한다. (회의 만들기 화면의 `DotWave`/`valueSlot` 패턴과 동일)
   1. **공통 피하고 싶은 시간대**(예: 점심) — 시작~종료 시간을 **여러 개** 추가. → `모든 요일 공통적으로 …는 피하고 싶어요.` (`avoid`, 모든 날짜)
   2. **공통 선호 시간대** — 시간대 여러 개. → `…는 선호해요!` (`preferred`, 모든 날짜)
   3. **불가능한 날짜** — `날짜 선택` 버튼을 누르면 달력 **모달**(`CalendarModalField`, 회의 만들기 `DatePicker` 모달과 같은 톤)이 열리고 거기서 **여러 날짜를 다중 선택**(빨강)한다. 선택한 날짜는 트리거 아래 칩으로 표시·삭제할 수 있다. → `…에는 회의가 불가능해요.` (`busy`, 하루)
   4. **선호하는 날짜** — 같은 방식의 날짜 모달에서 다중 선택(파랑). 앞 단계에서 **불가능으로 고른 날짜는 빨간색 비활성 + `불가` 배지**로 표시되어 선택할 수 없다. → `…는 선호해요!` (`preferred`, 하루)
   5. **특정 날짜+시간 불가** — `날짜 선택` 버튼 → 달력 **모달**에서 날짜를 고르면(단일 선택 후 닫힘) 그 날의 안 되는 시간대를 **여러 개** 추가한다. **앞서 불가능(busy)·선호(preferred)로 고른 날짜는 이 모달에서 선택할 수 없다**(그날의 가부가 이미 정해졌으므로). 이미 시간을 입력한 다른 날짜는 칩으로 빠르게 전환한다. → `… 에는 안 돼요.` (`busy`, 날짜+시간)
4. 마지막 단계에서 `응답 제출하기`로 제출한다.

달력(`MonthCalendar`)·시간 입력 공통 동작:

- 달력은 한 번에 **한 달만** 표시하고 `<` `>` 버튼으로 월을 이동한다(유효 날짜가 있는 월 범위 안에서만, 양 끝 월에서 버튼 비활성). 크기는 회의 만들기의 `DatePicker`와 비슷하게 컴팩트하다.
- **토·일요일은 비활성**으로 표시해 선택할 수 없다.
- 시간대 추가(피하고 싶은/선호/특정 날짜)는 네이티브 `time` 입력이 아니라 커스텀 `TimeSelect`(30분 단위 00:00~23:30 목록 모달)로 받는다. **근무시간(09:00~18:00) 밖 시간은 회색 + `근무시간 외` 표시**로 비추천하되, 선택하면 `… 근무시간을 벗어났습니다. 그래도 선택하시겠습니까?` **컨펌**을 거쳐 허용한다(허용된 값은 트리거에 주황색으로 표시). 기본 입력값은 근무시간 안에서 시작한다.
- 단, 회의 추천 후보는 09:00~18:00만 생성되므로 **완전히 근무시간 밖인 시간대는 저장 시 사실상 무시**된다(`buildBlocks`가 근무시간 행만 셀로 환산 → 걸친 경우 근무시간 부분만 저장). 즉 컨펌 후 선택은 되지만 추천에는 영향이 없는 입력이다.
- **선호 시간대**를 추가할 때 앞서 입력한 **피하고 싶은 시간대와 겹치면** 경고 토스트와 함께 추가를 막는다.

날짜/시간 상태 매핑:

- `preferred`: 선호 (선호 시간대 · 선호 날짜)
- `busy`: 불가능 (불가능 날짜 · 특정 날짜+시간)
- `avoid`: 피하고 싶음 (공통 피하고 싶은 시간대) — 확정 불가는 아니지만 추천에서 감점되는 약한 회피
- `available`: 기본값(아무것도 지정하지 않은 시간), DB 미저장

데이터 정책:

- 입력값(공통 시간대 + 불가/선호 날짜 + 특정 날짜+시간)을 우선순위로 셀(30분 단위)로 환산한 뒤 블록으로 저장한다. 우선순위(높은 것부터): 특정 날짜+시간(busy) > 불가능 날짜(busy) > 선호 날짜(preferred) > 공통 피하고 싶은(avoid) > 공통 선호(preferred).
- 제출 블록은 30분 단위로 정렬되며, **점심시간과 겹치는 셀은 제외**한다(서버가 점심 겹침 블록을 거부하므로).
- `available`은 저장하지 않는다.

제출 후 / 투표 UX:

- 응답 제출 후에는 일주일 그리드 회의 캘린더를 보여준다.
- 회의 캘린더는 날짜 컬럼 × 30분 시간 행 그리드이며, 화면을 넓게 쓰고 마우스 드래그·휠로 가로 스크롤한다. 시간(KST) 축은 좌측에 고정한다. 모든 시간 칸은 동일한 높이를 유지한다.
- 각 칸은 `선호`(초록)·`불가능`(빨강) 그룹으로 참석자 이름 pill 을 직접 표시하고, 이름 색은 참석자별로 다르다. 가능·미응답은 표시하지 않는다.
- 모든 참석자가 응답하기 전에는 대기 안내를, 응답이 모이면 추천 후보 목록을 보여주고 참석자는 후보 중 하나에 투표한다(재투표 시 교체).

수정 UX:

- 첫 제출 후 `participantToken`을 localStorage에 저장한다.
- 같은 브라우저로 다시 접속하면 본인확인 없이 제출 결과(회의 캘린더)를 바로 보여주고, `수정`을 누르면 가능 시간 입력을 처음부터 다시 받는다. (새 입력 모델이라 기존 블록을 역으로 채우지는 않는다.)
- 토큰이 없거나 일치하지 않으면 회의 안내(intro)부터 다시 진행한다.

## 5. UI 스타일 방향

modu는 한국어 협업 도구처럼 간결하고 신뢰감 있게 만든다.

디자인 원칙:

- 넓은 여백
- 명확한 카드 구조
- 큰 타이포그래피
- 부드러운 라운드
- 쉬운 문장
- 모바일/데스크톱 반응형
- 색상에만 의존하지 않는 상태 표현
- 페이지 전체·헤더 배경은 `rgb(255 255 255 / 0.95)`(Tailwind `bg-white/95`)로 통일한다. 단 **모바일 고정 헤더는 불투명 흰색(`bg-white`) + 하단 테두리 없음**으로 하고, 헤더 바로 아래를 흰색→투명 그라데이션(`.modu-mobile-header::after`)으로 흐려 본문과 부드럽게 잇는다. 데스크톱(헤더 static)에서는 그라데이션을 끈다.
- 모달은 화면 목적을 나타내는 타이틀만 두고, 불필요한 서브타이틀이나 설명 문구를 넣지 않는다.

토스 제품처럼 쉽고 명확한 인상을 참고하되, 토스 브랜드, 로고, 고유한 시각 요소를 직접 모방하지 않는다.

추천 카드가 가장 중요한 UI다. 화면 전체가 추천 후보를 빠르게 비교하는 데 집중되어야 한다.

## 7. 접근성 및 반응형 기준

- 모든 버튼은 명확한 텍스트 또는 접근 가능한 라벨을 가진다.
- 상태 색상에는 텍스트 라벨을 함께 붙인다.
- 시간표 셀은 키보드 포커스가 가능해야 한다.
- 모바일에서 텍스트가 버튼 밖으로 넘치지 않아야 한다.
- 모바일에서 `input`, `textarea`, `select`를 터치했을 때 iOS Safari 또는 Chrome이 화면을 자동 확대하지 않아야 한다. 입력 필드의 실제 `font-size`는 모바일에서도 16px 이상으로 유지하고, focus 시 강제 `scrollIntoView` 호출이나 viewport scale 조작을 넣지 않는다. 키보드는 올라오되 화면 확대나 입력 영역 잘림이 없어야 한다.
- 추천 카드의 핵심 정보는 작은 화면에서도 접히지 않고 먼저 보여야 한다.
- 모바일 주요 CTA 버튼(회의 만들기, 응답 제출 등)은 `MobileStickyAction`을 사용한다. 하단 영역은 테두리/그림자/블러 없이 본문 배경과 같은 불투명 배경을 유지하고, 본문과 맞닿는 위쪽 경계만 그라데이션으로 자연스럽게 이어지게 한다.
- 모바일 주요 CTA 때문에 짧은 화면에 가짜 스크롤이 생기면 안 된다. 실제 본문 내용이 화면 높이를 넘을 때만 본문 스크롤이 생겨야 한다.
- 복사 버튼, 확정 버튼 등 주요 액션은 터치하기 충분한 크기로 만든다.

## 8. 프론트엔드 테스트 시나리오

- 랜딩에서 회의 만들기로 이동할 수 있다.
- 모바일 랜딩에서 콘텐츠가 짧을 때는 본문 스크롤이 생기지 않는다.
- 회의 생성 폼에 기본값이 올바르게 들어간다.
- 회의명 20글자, 안건 30글자, 장소 20글자를 넘으면 라벨이 아이콘 포함 경고 문구로 바뀌고 입력창 테두리가 빨간색으로 바뀐다.
- 가상 직원 목록에서 참석자를 검색하고 선택/해제할 수 있다.
- 참석자는 최대 8명까지만 선택할 수 있다.
- 참석자를 1명 이하로 둔 채 `선택 완료`를 누르면 모달이 닫히지 않고 유효성 경고가 보인다.
- 선택된 참석자 뱃지를 필수참석/선택참석 구역으로 드래그해 참석유형을 변경할 수 있다.
- 공유 화면에서 참석자 링크를 복사할 수 있다.
- 참석자 링크 진입 시 회의 안내(intro) → `시간 정하러 가기` → 본인확인(이름·직무, 오류 시 토스트) 순으로 진행한다.
- 가능 시간 5단계(공통 피하고 싶은 시간대 → 공통 선호 시간대 → 불가능 날짜 → 선호 날짜 → 특정 날짜+시간)를 입력하고 제출할 수 있다. 시간대는 여러 개 추가할 수 있고, 날짜는 달력에서 다중 선택한다.
- 불가능/선호 날짜는 `날짜 선택` 버튼 → 달력 모달에서 다중 선택하며, 한 달씩 표시되고 `<` `>`로 월을 이동하고, 토·일과 (선호 달력에서) 불가능으로 고른 날짜는 선택할 수 없다.
- 근무시간 밖 시간은 회색 + `근무시간 외`로 표시되고 컨펌을 거쳐 선택할 수 있으며, 피하고 싶은 시간과 겹치는 선호 시간대는 경고와 함께 추가되지 않는다.
- 같은 브라우저에서 제출 후 다시 접속하면 결과 캘린더를 바로 보고 `수정`으로 다시 입력할 수 있다.
- 제출 후 참석자 화면에서 일주일 그리드 캘린더가 표시된다.
- 캘린더 각 칸에 선호(초록)·불가능(빨강) 참석자 이름이 직접 표시되고, 상세 메모 원문은 보이지 않는다.
- admin 화면에서 응답자와 미응답자를 구분해 볼 수 있다.
- admin 화면에서 전체 캘린더로 선호/불가능 집계를 볼 수 있다.
- 추천 후보 카드가 최소 3개 표시된다.
- 필수/선택 변경 시 추천 결과가 다시 계산된다.
- 모든 참석자가 응답하기 전에는 후보 투표가 열리지 않는다.
- 모든 참석자가 후보에 투표하면 최다 득표 후보를 확정할 수 있다. 1위가 동률이면 주최자가 동률 후보 중 하나를 고른다.
- 확정 화면에서 공유 문구 복사와 `.ics` 다운로드가 가능하다.

### 9.2 두 개의 "문장 빌더" 상태 머신

두 큰 폼은 같은 패턴(상단 = 쌓이는 한국어 문장, 하단 = 현재 단계 입력 + CTA)을 쓴다.

- **`MeetingCreateForm`** — 단일 `step`(0 회의명 → 1 안건 → 2 장소 → 3 마감일 → 4 진행시간 → 5 참석자). hidden input으로 전체 값을 항상 직렬화해 Server Action에 제출.
- **`ResponseForm`** — 2단계 머신:
  - `step`: `loading | intro | identity | availability | done`
  - `identity`의 `formStep`: 0 이름 → 1 직무(Select)
  - `availability`의 `availStep`: 0 피하고싶은시간 → 1 선호시간 → 2 불가날짜 → 3 선호날짜 → 4 특정날짜+시간 (`maxAvailStep`로 도달 단계 추적)
  - 제출 후 `done` → `SubmittedCalendarScreen`(현재 더미 6명 표시 — 실데이터 아님) + `VotingPanel`

### 9.4 핵심 규칙·상수

- 시간은 전부 **KST(고정 +09:00)**, 근무시간 **09:00~18:00 고정**, 슬롯/셀 **30분**.
- 상태색에는 **항상 텍스트 라벨**을 함께 붙인다(색만으로 상태 전달 금지).
- 모든 Supabase 접근은 **서버 전용**(`server-only`), service_role로 RLS 우회 → 권한은 앱 내부 **토큰 비교**(`admin_token`/`participant_token`)에 의존. 제품 용어 "주최자" = 코드의 `admin*`.
- 입력 한도는 `lib/meetingLimits.ts`(제목 20 / 안건 30 / 장소 20자, 참석자 2~8명).

### 9.5 함정 / 주의

- `ResponseForm.tsx`(1450+줄)·`MeetingCreateForm.tsx`(890+줄)는 한 파일에 하위 컴포넌트 다수. 수정 전 파일 내 정의 위치부터 grep.
- 가능 시간 `buildBlocks`는 **근무시간 행만** 셀로 환산 → 완전히 근무시간 밖인 시간대는 저장 시 무시됨(§4.4 참고).
- 제출 후 `SubmittedCalendarScreen`은 현재 **더미 6명(`MOCK_EMPLOYEES`)** 을 보여준다(실제 응답 아님, `buildDummySnapshot`).
- Supabase env 없으면 `lib/demoMeeting.ts` **데모 플로우**로 빠진다(제출 미저장·확정 불가). 로컬 개발은 `.modu-local-db.json` 파일 폴백.
- 점심 비활성화는 `lunch_start='00:00'/lunch_end='00:01'` 센티넬로 저장(서버가 점심 겹침 블록을 거부).
