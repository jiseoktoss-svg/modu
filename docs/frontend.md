# modu 프론트엔드 구현 문서

## 1. 프론트엔드 목표

modu의 프론트엔드는 사용자가 30초 안에 서비스 가치를 이해하고, 복잡한 일정 조율 과정을 부담 없이 완료할 수 있도록 구성한다.

핵심 UX 목표:

- 회의 생성자는 빠르게 회의를 만들고 링크를 복사할 수 있다.
- 참석자는 로그인 없이 모바일에서도 쉽게 시간표를 입력할 수 있다.
- 열람 조건이 충족되면 modu가 응답 분포를 해석해 캘린더에서 먼저 볼 날과 피해야 할 날을 정리해 보여준다(투표도 자동 확정도 없음 — 최종 회의 시간은 참여자들이 제품 밖에서 정한다).
- 회의 생성자와 참석자는 같은 화면에서 전체 캘린더, 날짜별 상대 추천도, 날짜·시간 조회 결과를 이해할 수 있다.
- 제품 안에는 관리자 개념, 관리자 권한, 관리자 전용 페이지를 두지 않는다.
- 모든 상태는 색상뿐 아니라 텍스트 라벨로도 표현한다.

## 2. 기술 기준

- Next.js App Router
- TypeScript
- Tailwind CSS
- Server Components 기본 사용
- 상호작용이 필요한 폼, 시간표, 복사 버튼, 결과 캘린더/후보 재계산 UI는 Client Components로 분리
- 본문 텍스트 폰트는 `Pretendard`(globals.css의 CDN `@import` + `tailwind.config.ts`의 `fontFamily.sans`에 반영)
- 별도 아이콘 라이브러리를 두지 않는다. UI 아이콘은 유니코드 이모지를 `components/ui/Emoji.tsx`(`TossFace` 폰트)로 렌더링한다. (`lucide-react` 의존성 제거)
- 이모지는 `public/fonts/TossFaceFontWeb.otf`(COLR/CPAL, 전 브라우저 컬러)를 우선 사용하고 `public/fonts/TossFaceFontMac.ttf`로 폴백하며, `TossFace` 폰트의 `unicode-range`로 이모지 코드포인트에만 적용된다
- 디자인/컴포넌트는 Tailwind CSS 기반의 프로젝트 내부 컴포넌트를 사용한다. 버튼, 입력창, 카드, 배지, 날짜 선택기, 복사 버튼 등은 `components/ui/` 아래 자체 컴포넌트로 구성한다.
- `next.config.mjs`는 기본 Next.js 설정을 유지한다. 별도 UI 라이브러리 번들 최적화 설정은 두지 않으며, Turbopack(`next dev --turbo`)도 켜지 않는다.
- 토스트 알림은 앱 공통 패턴(상단 고정 pill + 아이콘 + 자동 사라짐)으로 제공한다. (예: 참석자 본인확인 검증 오류, 링크 복사 완료) 모든 토스트는 `회의가 만들어졌어요` 화면의 `복사 완료` 토스트와 **같은 크기·radius**(`rounded-[16px]`, 내용 폭에 맞춘 `inline-flex` pill)로 통일하고, **색상까지** 모두 흰 배경(`bg-white`)·진한 글자(`text-slate-800`)로 통일한다(용도와 무관하게 `복사 완료` 토스트와 동일). PC·모바일 동일하며, 문구는 짧게 유지한다. 등장·사라짐 애니메이션도 `복사 완료` 토스트와 동일하게 맞춘다(요소는 계속 렌더하고 `open` 클래스만 토글, 사라지는 동안 문구를 유지해 부드럽게 페이드아웃).
- 모든 버튼(`TDSButton`)은 누름(press) 피드백을 준다: 모바일 탭·데스크톱 클릭 모두 `:active` 시 살짝 축소(`scale(0.97)`)되고, 데스크톱은 호버 시 명도가 살짝 낮아진다(저감 모션 시 비활성). 버튼 로딩 표시는 점 3개 파도타기(`animate-dot-wave`, 글자색)로 통일한다.
- 모바일 하단 입력 영역과 CTA는 `MobileStickyAction`을 재사용한다. 상단 내용이 길어져 스크롤이 생겨도 하단 입력/CTA 위치는 유지하며, 하단 영역 자체는 불투명한 본문 배경색을 유지한다. 본문과 맞닿는 부분만 선·그림자 없이 흰색 그라데이션으로 부드럽게 이어지게 한다.
- `MobileStickyAction`은 모바일에서 하단 고정 영역 높이 + 약간의 여백(20px)만큼 본문 아래 공간을 **항상 일정하게** 확보한다(`window.innerHeight`에 의존하지 않음). 주소창이 열리고 닫혀도 스페이서 높이가 변하지 않아, 스크롤 도중 위치가 튀거나 손을 떼면 원래대로 되돌아가는 현상이 없다. 본문이 길면 자연스럽게 스크롤되며 마지막 내용과 하단 영역 사이 여백이 유지된다. 짧은 화면(flex 레이아웃)에서는 본문이 늘어나 채우므로 가짜 스크롤이 생기지 않는다.

## 4. 화면별 구현 상세

### 4.1 랜딩 화면

필수 콘텐츠:

- 서비스명: `modu` (헤더 로고로 노출 — 히어로 영역과 메인/서브 카피는 제거됨)
- 데모 안내(고정 CTA 버튼 바로 위 작은 회색 글씨): `지금은 데모로 열려 있어요 — 마음껏 눌러보세요`
- CTA: `회의 만들기` — **PC·모바일 공통 하단 고정(fixed)**. 위쪽은 흰색 그라데이션으로 본문과 잇고, 버튼 폭은 다른 페이지 CTA와 동일(본문 컬럼 `max-w-2xl` + `sm:px-6`). CTA 아래에는 아무것도 두지 않는다. **인트로 모션 재생이 끝나야 페이드인**된다(재생 중에는 숨김 + `pointer-events-none`). **건너뛰기·다시 보기 버튼은 두지 않는다.**

구성(인트로 모션, `components/landing/LandingIntro.tsx`):

- 기존 3단 문구|이미지 섹션을 **약 15.1초짜리 4장면 인트로 모션**으로 교체했다(Claude Design 시안 이식, 외부 라이브러리·영상 파일 없이 CSS keyframes + 인라인 SVG). 구버전 `Reveal.tsx`/`LandingIllustrations.tsx`/`public/landing/*-clay.png`는 삭제(git 히스토리 참고).
- **장면 캡션이 이어지는 한 문장을 이룬다**(문장 빌더 정체성): ①`어려운 시간만 알려주면 돼요`(카드 안) → ②`모두의 응답을 모아 분석하고` → ③`불가능한 날은 제외하여` → ④`최적의 시간을 알려드려요` → 워드마크 `가장 나은 시간을 찾아드려요`가 마침표. ②~③은 캔버스 하단 중앙(`SceneCaption`, bottom 24)에 뜨고, ④는 마지막 후보 리스트 화면 아래쪽에 배치해 후보 행과 겹치지 않게 한다.
- 장면 1 (0~3.6s) **응답 수집**: 참석자 응답 카드 클로즈업 — 안내 문구 `어려운 시간만 알려주면 돼요`가 입력창 위에 뜨고, 문장이 한 글자씩 타이핑되며(불가 값은 빨간 강조, 글자가 폭을 갖고 늘어나는 `mi-char-type`이라 **캐럿이 입력 위치를 따라간다**) 커서가 **일정한 속도로**(0.9s linear + 거리 비례 구간 배분) 이동해 `응답하기`(파란 그라디언트 버튼) 위에 오면 **그라디언트 테두리 글로우 호버 효과**(≈2.9s)가 켜진 뒤 클릭한다.
- 장면 2 (3.6~8.1s) **해석**: 참석자 뱃지 6명(필수 4·선택 2, 필수 배지는 그라디언트) — 세로 중앙(분석 필 자리)은 비워 위/중간/아래 세 줄로 배치 — 에서 나온 빨간 불가 칩들이 비슷한 거리로 중앙에 수렴하고, 그라디언트 테두리가 흐르는 `분석 중…` 필 + 글로우가 하이라이트를 만든다(스파클 아이콘 없음).
- 장면 3 (8.1~12.25s) **선별**: 날짜 숫자·테두리 없이 칸만 있는 온전한 7월 달력(타이틀은 캘린더 아이콘)이 먼저 보이고 → 날짜칸 위를 **흰빛 스캔 광선**(`mi-scan`, 8.7s~, 1.5s로 느리게)이 한 번 훑고 지나가면 → 탈락한 날이 **빨갛게 물들고**(`mi-cell-mark`, 10.15s~) → 순서대로 좌우로 기울며 우수수 낙하한다(`mi-cell-fall-l/r`, 10.55s~).
- 장면 4 (11.9s~) **후보 리스트**: 새 화면에서 **후보 리스트 12줄**(날짜만, 배지 없음, 행마다 다른 연블루 그라디언트 배경)이 순차 등장했다가 캡션과 함께 **위로 빠르게 날아가며**(`mi-list-up`, 13.4s~) 사라지고, 팡 터지는 버스트(`mi-burst`)와 함께 그라디언트 `modu` 워드마크가 팝(`mi-pop-in`, 14.15s)으로 등장한다. 종료(15.1s) 시 CTA가 페이드인된다.
- 캔버스(무대)는 `rounded-3xl` 흰색 카드로, **헤더와 하단 CTA 사이 남은 세로 공간을 flex(`flex-1`)로 꽉 채운다**(고정 높이 아님). 내부 좌표는 `340×500`(`ARTBOARD_W`/`ARTBOARD_H`) 고정 아트보드이며, **무대 크기에 맞춰 비율 그대로 축소(`transform: scale`, 상한 1)** 해 작은 폰에서도 모션 전체가 스크롤·잘림 없이 들어온다. 배율은 무대 `clientWidth/Height`를 `ResizeObserver`로 재서 `min(w/(340+ARTBOARD_BLEED·2), h/500, 1)`로 구하고(주소창 열림/닫힘·회전에 재계산 — 폭 항 분모에 `ARTBOARD_BLEED`(8px·좌우)를 더하는 건 장면 2 배지가 아트보드 밖으로 8px씩 삐져나와 폭이 빡빡한 세로 긴 기기에서 `overflow-hidden`에 잘리는 걸 막기 위함), 하단 CTA 실제 높이(`ResizeObserver`)만큼 본문 `paddingBottom`을 잡아 무대가 CTA에 가리지 않게 한다. **아트보드 내부는 px 좌표·딜레이가 맞물려 있어 Tailwind 대신 인라인 스타일을 유지**한다. `mi-*` keyframes 는 컴포넌트 안 `<style>` 태그에 있다.
- 동작 제어: **새로고침을 포함해 페이지가 열릴 때마다 처음부터 재생**한다(재방문 스킵 없음). `prefers-reduced-motion`만 애니메이션 없이 마지막 장면+CTA를 바로 보여준다. SSR 시에는 빈 캔버스(`pending`)만 렌더해 hydration 불일치를 막는다.
- 문구 말투: 해요체·능동·긍정, {명사}+{명사} 조합은 쓰지 않고 한자어 명사는 동사로 풀어 쓴다. (전체 규칙: `docs/ux-writing.md`)

화면 방향:

- 첫 화면에서 제품명(헤더 로고)과 인트로 모션이 분명히 보여야 한다.
- 인트로만 봐도 단순 투표 앱이 아니라 의사결정 도구라는 점이 드러나야 한다(수집→해석→선별 서사).
- 마케팅 페이지처럼 과하게 장식하지 않고, 실제 서비스 진입을 우선한다. 페이지가 열릴 때마다 인트로를 끝까지 본 뒤 CTA 가 열린다.
- **모든 페이지 래퍼와 `body`는 `min-h-screen`(=100vh)이 아니라 `min-h-dvh`를 쓴다.** 모바일에서 `100vh`는 주소창이 숨겨진 큰 뷰포트 기준이라 항상 약간의 팬텀 스크롤이 생기고, 스크롤·입력 시 주소창이 사라지며 본문이 위로 튀어 오른다. `dvh`(동적 뷰포트)로 잡아 이 자동 스크롤을 없앤다. **단 랜딩(`app/page.tsx`)만 예외로 `h-dvh overflow-hidden`(정확히 한 화면)을 써서 스크롤을 원천 차단**하고, 넘칠 수 있는 모션은 위 아트보드 축소로 흡수한다.

### 4.2 회의 만들기 화면

주요 컴포넌트:

- `MeetingCreateForm`
- `ParticipantListEditor`

화면 톤:

- 페이지 배경은 `rgb(255 255 255 / 0.95)`(Tailwind `bg-white/95`, 모든 페이지·헤더 공통)이며, 카드 없이 라벨 텍스트 + 입력창으로 구성한다.
- 폼 내부 상단 여백은 모바일 `pt-4`(16px) / 데스크톱 `sm:pt-8`로 둔다(참석자·확정 화면과 모바일 상단 여백 통일).

폼 기본값:

- 회의 길이: 1시간 0분
- 회의 마감 날짜: 비어 있는 상태로 시작(자동 선택하지 않음). 고르기 전까지 버튼에 `날짜를 선택해주세요`를 표시한다. 최소 선택일은 오늘+2일(응답 마감일이 회의 마감 2일 전까지여야 하므로).
- 응답 마감일: 날짜는 비어 있는 상태로 시작(자동 선택하지 않음), 시각은 18시 기본. 선택 가능 범위는 오늘 ~ 회의 마감 날짜 −2일.
- 내부 계산용 근무 시간: 09:00~18:00
- 신규 회의의 날짜 범위: 오늘부터 회의 마감 날짜까지
- 참석자: 비어 있는 상태로 시작

문장 빌더 UX(한 페이지 안의 단계형):

- 회의 생성자가 항목을 하나씩 입력할수록 페이지 상단의 한국어 안내 문장이 완성되고, 하단에는 현재 항목 입력창과 `다음` 버튼이 위치한다.
- 상단 문장의 값을 클릭하면 그 항목으로 돌아가 수정할 수 있다.
- 아직 채우지 않았거나 지운 값 자리에는 회색 점 3개의 `파도타기`(dot-wave) 애니메이션을 표시한다.
- 새 절/단계는 **아래에서 위로 떠오르며 흐릿→뚜렷해지는 AOS 애니메이션**(`fade-up-blur`)으로 등장한다. (인라인 절이라 `transform` 대신 `position:relative`+`top` 기반) 하단 입력창에는 등장 애니메이션을 두지 않는다.
- 순서: 회의명 → 안건 → 장소 → 예상 회의 진행 시간(시간/분) → 회의 마감 날짜 → 응답 마감일(날짜+시각) → 참석자
- 응답 마감일 단계에서 `다음`을 누르면 참석자 단계로만 이동한다. 참석자 선택 모달은 참석자 단계의 `참석자 선택하기` 버튼을 눌렀을 때 열린다. 선택 후 상단 문장에 참석자 이름과 총 인원을 표시한다.
- 상단 안내 문장 영역은 좌측 정렬이다(참석자 명단 포함).
- 최종 문장 형태: `이번 회의명은 {회의명}에요. 회의 안건은 {안건} 입니다. 회의 장소는 {장소}이며, 예상 회의 진행 시간은 {시간}시간 {분} 분입니다. {날짜(요일 포함, 예: 2026년 7월 5일 일요일)}까지는 회의가 완료되어야 해요. 참여자는 {응답 마감일(요일 포함) 시각}까지 응답해주세요.` 그리고 줄을 바꿔 `회의 참석자 명단은 {이름1}, {이름2}, … 총 N명 입니다.` (시간 값이 0이면 시간 부분, 분 값이 0이면 분 부분을 문장에서 생략한다.)

입력 컨트롤:

- 회의명은 공백 포함 최대 20글자, 회의 안건은 공백 포함 최대 30글자, 회의 장소는 공백 포함 최대 20글자까지 입력할 수 있다.
- 글자 수 제한을 넘으면 기존 입력 라벨이 `!` 아이콘 + 빨간 경고 문구로 바뀌고, 입력창 테두리도 같은 빨간색으로 바뀐다. 별도 경고 문구를 라벨 아래에 추가하지 않는다.
- 입력창 포커스 테두리는 `#4593fc`(Tailwind `brand-400`) 단일 색상으로 통일하고, 포커스 시 테두리 두께만 1px 더 두껍게 한다. 별도 파란 링은 사용하지 않는다.
- 회의 마감 날짜는 디자인 톤에 맞춘 커스텀 캘린더 `DatePicker`(요일 표시, KST 안전, `min=오늘+2일`)로 받는다. `input[type=date]`는 사용하지 않는다. 값이 없을 때 버튼에는 `날짜를 선택해주세요`를 표시한다(시작 시 자동 선택하지 않음). 최소를 오늘+2일로 둔 것은 응답 마감일이 회의 마감 2일 전까지여야 하기 때문이다.
- 응답 마감일은 같은 `DatePicker`(날짜) + 시(0~23시) `Select`로 받는다. **시 `Select`는 네이티브 `<select>`가 아니라 `variant="menu"` 커스텀 포털 드롭다운**(`MenuSelect`, 뷰포트 인지·키보드/Esc/바깥클릭 지원)이다. **분 단위는 받지 않는다(항상 :00).** 응답 마감일은 **회의 마감 날짜 2일 전(`회의 마감일 −2일`)까지만** 고를 수 있고, 그 이후 날짜는 `DatePicker`의 `max`로 **비활성화**(회색 + `다음 달` 이동 버튼도 차단)되며 비활성 날짜에 마우스를 올리면 사유 툴팁(`minReason`/`maxReason`, 포털로 표시)을 보여준다. 범위를 벗어난 값을 고르거나 회의 마감일을 더 앞당기면 인라인 경고 대신 **경고 토스트**를 띄우고 값을 회의 마감 날짜 −2일로 자동 보정한다.
- `DatePicker` 달력은 화면 중앙 모달로 띄운다. 모달은 여유 있는 크기·내부 패딩으로 두되 작은 모바일 화면에서도 화면 밖으로 넘치지 않게 한다. 닫기 버튼은 `h-11 w-11`(아이콘 20)로 충분히 크게 둔다.
- 회의 길이 입력 라벨은 `예상 회의 진행 시간을 입력해주세요`이며, 시간/분은 프로젝트 내부 숫자 스테퍼(+/− 버튼과 직접 입력)로 받는다. 시간/분 입력 영역은 좌측 정렬이다.
- 시간과 분이 모두 0이면 라벨 아래에 `적절한 시간값을 입력해주세요.` 안내를, 9시간 초과 시 근무시간 안내 문구를 표시한다.
- `Enter` 키는 다음 단계로 진행만 하고 폼을 제출하지 않는다. 참석자 선택을 완료(2명 이상 + 모달 닫음)하면 마지막 단계 CTA가 `다음`이 되어 **회의 확인 화면**으로 넘어가고, 실제 제출은 그 화면의 `회의 만들기` 버튼으로만 한다.

검증(모든 항목 필수):

- 회의명, 안건, 장소, 예상 회의 진행 시간, 회의 마감 날짜, 응답 마감일이 모두 필수다.
- 회의명은 최대 20글자, 안건은 최대 30글자, 장소는 최대 20글자까지 허용한다.
- 참석자는 최소 2명 이상, 최대 8명까지 허용한다.
- 회의 길이는 시간과 분을 직접 입력하며, 합산 시간이 근무 시간(9시간) 이내여야 한다.

회의 생성자는 근무 시작/종료 시간과 점심 시간을 입력하지 않는다. 화면에서는 회의 마감 날짜와 응답 마감일을 받고, 서버가 오늘부터 마감 날짜까지의 범위와 09:00~18:00 고정 근무 시간을 저장한다.

참석자 모달:

- 모바일에서는 참석자 선택 모달을 전체 화면으로 띄우고, 내용은 가능한 한 한 화면 안에서 보이도록 컴팩트하게 배치한다. 데스크톱에서는 고정 크기 + 내부 스크롤(내용이 늘어도 크기 유지)을 사용한다. 데스크톱은 직원 리스트 구역 높이(`sm:h-80`)와 모달 높이(`sm:h-[744px]`)를 넉넉히 두고, 모바일 레이아웃은 변경하지 않는다.
- 바깥(backdrop) 클릭이나 `Esc`로 닫고, 열릴 때 포커스를 이동하고 focus trap을 둔다.
- 모달 상단에는 `참석자 선택` 타이틀만 두고, 닫기 버튼은 프로젝트 내부 버튼 스타일을 쓴다. 별도 서브타이틀이나 설명 문구는 두지 않는다.
- 레이아웃은 위/아래 구성이다. 위에는 검색 입력창(왼쪽 정렬, 높이·상하 여백을 컴팩트하게 줄임)과 50명의 가상 직원 목록을 두고, 아래에는 `필수참석`·`선택참석` 두 구역을 둔다.
- 멤버 구역 아래에 별도 안내문은 두지 않는다. 대신 `선택 완료`를 참석자 1명 이하인 상태로 누르면 모달을 닫지 않고 `참석자는 최소 2명 이상 선택해 주세요.` 유효성 경고를 모달 안에 띄운다.
- 직원은 이름과 직책을 가지며, 이름 또는 직책으로 검색할 수 있다.
- 직원 목록 항목을 클릭하면 참석자를 추가/해제 토글한다. 참석자가 8명이 되면 아직 선택하지 않은 직원은 추가할 수 없게 비활성화한다.
- 선택된 참석자는 pill(둥근 뱃지) 형태로 아래 구역에 표시된다.
- 새로 추가하면 기본 `필수참석` 구역으로 들어간다(기본 `attendanceType=required`). 필수↔선택 이동은 뱃지 드래그 앤 드롭으로 한다.
- 빈 공간을 드래그하면 사각형(마퀴)으로 여러 명을 한 번에 선택한 뒤 함께 옮길 수 있다. 모바일에서는 선택된 참석자 pill을 터치 드래그해 필수참석↔선택참석 구역으로 옮길 수 있다.
- 직원 목록 영역은 위/아래에 페이드를 둔다.
- `필수참석`·`선택참석` 구역 제목 옆에는 인원 수만 옅게 표시한다. 별도의 `필수 N명 / 선택 N명` 인원수 pill 배지는 두지 않는다.
- `선택 완료`는 전체폭 버튼으로 표시한다.

회의 확인 화면:

- 폼을 모두 채우고 마지막 단계의 `다음`을 누르면 생성 직전 `회의 확인` 화면으로 전환한다(`MeetingCreateForm`의 `confirming` 상태). 상단에 작은 `회의 확인` 라벨, 그 아래 1~7번 문장이 **글자 하나씩 좌→우 잉크처럼 칠해지며 순차 등장**한다(`.modu-fill-char` mask+blur 스윕 — **모바일은 blur 없는 변형**(`modu-fill-char-reveal-flat`, filter 래스터화로 인한 키워드 흐림·미노출 글자 자글거림 방지), 대기 중 글자는 `opacity:0`으로 완전히 숨김. 이웃 글자와 살짝 겹치며 진행. 타이밍·슬롯 규칙은 `lib/charFill.ts` 공용 — 글자 시차 160ms·지속 300ms·절 사이 300ms, 공백 0/좁은 문장부호 0.5 슬롯). 채움이 모두 끝나면 mask 를 걷고 값(EditValue)에 intro 와 동일한 `modu-value-shine` 을 켠다. 날짜값(회의 마감·응답 마감)은 년도 없이 표시한다.
- 각 값은 폼 상단과 동일한 클릭 가능한 파란 키워드(`EditValue`)로, 누르면 확인 화면을 닫고 해당 입력 단계(참석자는 참석자 모달)로 돌아간다. 마지막 문장 아래에는 `수정하려면 키워드를 눌러 해당화면으로 이동하세요.` 안내를 둔다.
- 마지막 7번 문구가 등장한 뒤(`confirmCtaReady` 타이머)에야 하단 `회의 만들기` 버튼이 페이드인되며, 이 버튼(`SubmitButton`, `type=submit`)이 실제 `createMeeting`을 호출한다. 확인 화면을 벗어나면 타이머는 리셋된다.

### 4.3 회의 생성 완료 · 공유 화면

주요 컴포넌트:

- `MeetingCreatedPanel`(회의 생성 직후 `회의가 만들어졌어요` 화면)
- `ShareLinksPanel`, `CopyButton`

구성:

- 헤더는 **PC·모바일 모두 로고**를 노출한다(`<SiteHeader mobileLogo />`). 완료 화면이라 뒤로가기·화면 타이틀은 두지 않고 로고만 둔다(랜딩과 동일 패턴).
- 상단에 완료 애니메이션(체크 + 헤일로 + 버스트, CSS/SVG)과 `회의가 만들어졌어요` 타이틀. 완료 애니메이션은 즉시 보이고, 제목 → 요약 카드 → 링크 카드가 위에서부터 순차로 떠오른다(`fade-up-blur`, 약 120/260/400ms). 하단 CTA는 지연 없이 바로 노출한다.
- 회의 내용 요약 카드(회의명·안건·장소·마감일·예상 시간)와 참석자 명단(필수/선택 뱃지).
- `참석자 전달 링크` 카드 + 복사 버튼.
- 요약 카드와 링크 카드는 좌우 테두리 없이 상하 테두리만 둔다(`border-y`).
- 하단에는 `링크 복사` / `회의시간 입력하기` 2개 버튼을 균등 그리드(`grid-cols-2`)로 두고 `MobileStickyAction`(모바일 화면 하단 고정, 본문은 그 위로 스크롤)으로 배치한다. `회의시간 입력하기`는 참석자 응답 페이지(`/m/{id}`)를 **새 탭**으로 연다(`target="_blank"`). 기존 `수정하기` 텍스트 링크/버튼과 `editPath` 변수는 제거했다.
- 초기 노출 시 상단이 잘리지 않도록 세로 중앙정렬을 `my-auto` 대신 `[justify-content:safe_center]`로 둔다(이 화면은 세로 중앙정렬이라 모바일 상단 여백 통일 대상에서 제외).

복사 UX:

- 복사 성공 시 상단에 `복사 완료` 토스트(아이콘 포함)를 잠시 표시한다.
- 링크는 잘리지 않게 한 줄 말줄임으로 보여준다.

반응형:

- 모바일에서는 참석자 명단을 **3열 그리드(뱃지 폭을 줄여 한 줄에 3명)**, 콘텐츠를 세로 가운데로 모아 링크 카드 위아래 여백을 맞추고 스크롤 없이 한 화면에 담는다. 데스크톱에서는 명단을 가로 wrap으로 더 높게 둔다.

### 4.4 참석자 응답 화면

링크를 받은 참석자는 회의 만들기 화면과 같은 "문장 빌더" 톤으로 응답한다. 하단에서 한 단계씩 입력할 때마다 상단에 한국어 문장이 쌓이고, 상단 문장의 값을 클릭하면 해당 단계로 돌아가 수정할 수 있다. (`ResponseForm`) 입력은 **언제 되는지가 아니라 언제 안 되는지(불가)** 만 받는 "불가 중심" 모델이다.

`step` 머신: `loading | intro | identity | availability | review | waiting | result | done`.

흐름:

1. **회의 안내(intro)**: 링크 진입 시 회의 내용을 문장으로 보여주고(`MeetingSummarySentence`, 참석자 이름·명수는 제외, 날짜값은 년도 없이 표시, 값은 `font-bold` 파란 강조), 하단 `시간 정하러 가기` 버튼으로 본인확인 화면으로 이동한다. 응답 마감일이 있으면 문장에 `응답은 {일시}까지 부탁드려요`를 포함한다. (상단 문장은 회의 확인 화면과 동일한 **글자 잉크 채움**(`fill` prop, `lib/charFill.ts` 공용 타이밍)으로 등장하고, 채움이 끝나면 값에 은은한 shine 효과 `modu-value-shine` 이 켜진다. `시간 정하러 가기` CTA 도 회의 확인 화면처럼 **채움이 끝난 뒤에야**(`onFillDone` → `introCtaReady`) 페이드인된다)
2. **본인확인(identity)**: 이름 → 직무(Select)를 한 단계씩 입력하는 문장 빌더(`저는 ○○이고, 직무는 ○○이에요.`). **직무 `Select`는 네이티브 `<select>`가 아니라 `variant="menu"` 커스텀 포털 드롭다운**(`MenuSelect`)이다. `확인하고 다음`을 누르면 명단(roster)과 대조해 검증하고, **검증 오류는 아이콘 토스트**로 알린다. 성공하면 `✅ 본인 확인 완료` 토스트를 띄우며 다음(가능 시간) 화면으로 넘어간다. `이름을 입력해주세요` 라벨 옆 `?` 아이콘(`HelpTooltip`, 클릭 토글 — hover 가 아니라 모바일에서도 동작)을 누르면 **본인확인을 하는 이유**를 툴팁으로 보여준다. 툴팁은 **모바일은 portal + `fixed inset-x-4`로 뷰포트 이탈을 막고, 데스크톱은 `left-0` 앵커 + 고정폭(`w-[25rem]`)으로 2줄 표현**해 좌우 이탈을 해결한다.
3. **가능 시간(availability — 불가 중심 2단계 문장 빌더)**: 하단 질문에 답할 때마다 상단 문장이 이어진다. 단계마다 입력이 없어도 건너뛸 수 있다. 상단 문장에서 **아직 정하지 않은(현재 진행 중) 항목은 회색 점 3개 `파도타기`(dot-wave)** 로 표시하고, 지나온 단계는 답(`…없고`/`…없어요` 포함)으로 확정 표시한다. (회의 만들기 화면의 `DotWave`/`valueSlot` 패턴과 동일) 확정값은 모두 부정(불가)이라 **연한 빨강**(`EditValue` `tone="negative"`, `text-red-400`)으로 강조한다. 가능 시간 단계의 질문·입력란은 페이지 하단에 배치한다(`mt-auto`).
   1. **불가능한 날짜** — `날짜 선택` 버튼을 누르면 달력 **모달**(`CalendarModalField` → 공용 `CalendarModal`)에서 **여러 날짜를 다중 선택**(빨강)하고 `확인`(배경 탭·✕ 도 가능)으로 닫는다. 모달은 공용 `components/scheduler/CalendarModal.tsx`로 **모바일은 전체화면(상/하단 테두리 없음 + 달력 좌우 꽉참 + 하단에 선택 날짜 칩), 데스크톱은 가운데 카드**(회의 만들기 `DatePicker`와 시각적으로 일치)로 뜬다. 선택한 날짜는 트리거 아래 칩으로도 표시·삭제할 수 있다. (`busy`, 하루 전체)
   2. **특정 날짜+시간 불가** — 같은 공용 `CalendarModal`을 쓴다. **모바일**은 전체화면 모달 안에서 **달력으로 날짜를 고르면 그 아래(모달 `extra` 영역)에 시간 입력기가 나타나** 그 날의 안 되는 시간대를 **여러 개** 추가하고, **모달 안에 `지금까지 추가한 안 되는 시간` 누적 목록을 렌더해 다른 날짜를 골라도 기존 추가분이 계속 보인다.** `확인`으로 닫는다(날짜+시간을 한 창에서 함께 입력). **데스크톱**은 이전 방식대로 가운데 카드에서 날짜만 고르면 닫히고, **페이지 아래에서 그 날의 시간대를 추가**한다. 입력한 날은 `MM/DD 시간들` 요약 벳지로 표시하고(벳지 탭=수정, ✕=그 날 전체 삭제) 다른 날로 전환한다. **PC는 시간 입력 카드 안에 개별 시간 벳지를 두지 않고(모바일과 동일하게 `renderTimeAdder`의 `showChips=false`), 카드 아래 요약 벳지만 한 번 노출한다.** **앞서 불가능(busy)으로 고른 날짜는 이 달력에서 선택할 수 없다**(`불가` 배지 + 비활성). (`busy`, 날짜+시간)
4. `다음`을 누르면 **입력 확인(review)** 화면으로 간다.

상단 문장(2절): `저는 {불가능한 날짜}에는 회의가 불가능하고, 특별히 {특정 날짜+시간}에는 안 돼요.` 값이 없으면 각각 `불가능한 날짜는 없고,` · `특정 시간에 안 되는 날은 없어요.`로 표시한다. (제거됨: 공통 피하고 싶은 시간·공통 선호 시간·선호 날짜 — 더 이상 받지 않는다.)

달력(`CalendarModal` 내부 `CalendarGrid`)·시간 입력 공통 동작:

- 달력은 한 번에 **한 달만** 표시하고 `<` `>` 버튼으로 월을 이동한다(유효 날짜가 있는 월 범위 안에서만, 양 끝 월에서 버튼 비활성). 크기는 회의 만들기의 `DatePicker`와 비슷하게 컴팩트하다(모바일 전체화면에서는 좌우로 꽉 채운다).
- **토·일요일은 비활성**으로 표시해 선택할 수 없다.
- 특정 날짜의 안 되는 시간대 추가는 네이티브 `time` 입력이 아니라 커스텀 `TimeSelect`(30분 단위 00:00~23:30 목록 모달)로 받는다. **근무시간(09:00~18:00) 밖 시간은 회색 + `근무시간 외` 표시**로 권장하지 않되, 선택하면 `… 근무시간을 벗어났습니다. 그래도 선택하시겠습니까?` **컨펌**을 거쳐 허용한다(허용된 값은 트리거에 주황색으로 표시). 시간 입력 **기본값은 시작 오전 09:00 / 종료 오전 10:00(1시간 창)**이다(근무 시작 = 시작, 시작+1시간 = 종료). 추가한 시간대는 칩(뱃지)으로 표시된다.
- 단, 회의 후보 슬롯은 09:00~18:00만 생성되므로 **완전히 근무시간 밖인 시간대는 저장 시 사실상 무시**된다(`buildBlocks`가 근무시간 행만 셀로 환산 → 걸친 경우 근무시간 부분만 저장). 즉 컨펌 후 선택은 되지만 결과 캘린더 집계에는 영향이 없는 입력이다.
- 같은 날짜에 **이미 추가한 것과 동일한 시간대(시작·종료가 같은 값)는 중복 추가를 막는다**(경고 토스트). 시작이 종료보다 늦어도 경고 토스트로 막는다. 추가된 시간대·날짜 칩의 삭제(X) 아이콘은 칩 색(빨강 텍스트 색)을 그대로 따른다.
- 달력 모달의 컨테이너는 **바깥 테두리 없이** 그림자만으로 구분한다(공용 `CalendarModal` 데스크톱 카드·`DatePicker` 모달 공통).
- **모바일 전체화면 시트의 하단 CTA 경계**: 본문과 CTA 사이를 흰색 그라데이션으로 잇고(`MobileStickyAction`과 동일 패턴), 추가/변경한 요약 칩이 스크롤 밖(아래)에 가려져 있으면 **하단 경계 전체가 tone 색으로 깜빡인다**(`ctaGlow` prop + `.modu-cta-glow`, 1.6초 주기). 스크롤해서 칩이 절반 이상 보이면 확인된 것으로 빼고, 미확인 칩이 하나도 남지 않으면 불빛이 꺼진다. 저감 모션에서는 깜빡임 없이 은은하게 켜진 상태로 고정된다.

날짜/시간 상태 매핑:

- `busy`: 불가능 (불가능 날짜 · 특정 날짜+시간) — `buildBlocks`가 저장하는 유일한 상태.
- `available`: 기본값(불가로 지정하지 않은 모든 시간), DB 미저장.

데이터 정책:

- 불가 입력(불가능 날짜 + 특정 날짜+시간)을 셀(30분 단위)로 환산해 `busy` 블록만 저장한다. `buildBlocks`는 불가 날짜(하루 전체)·특정 날짜+시간에 걸친 셀만 `busy`로 표시한다.
- 제출 블록은 30분 단위로 정렬되며, **점심시간과 겹치는 셀은 제외**한다(서버가 점심 겹침 블록을 거부하므로).
- `available`은 저장하지 않는다.

제출(review) → 대기(waiting) → 캘린더(done):

- **review**: 입력을 한 번 더 문장으로 보여주고 `다음`으로 제출(`submitAvailability`)한다. 문장은 `{이름}님은 …` 으로 시작하고 회의 확인 화면과 동일한 **글자 잉크 채움**(`.modu-fill-char`, `lib/charFill.ts` 공용 타이밍)으로 등장하며, `다음` CTA 도 채움이 끝난 뒤에야 페이드인된다. 필드값은 **좌측정렬**(`EditValue` inline `text-left`), 색은 채움 중에는 `red-400`, 채움이 끝나면 이전 화면과 같은 **연한 빨강 shine**(`.modu-value-shine-red`, `red-400` 기반)으로 강조한다. 하단 안내 문구는 `응답 시간 마감 전까지 수정할 수 있어요. 수정하려면 키워드를 눌러 응답화면으로 이동하세요.`이며(**별도 `수정하기` 버튼은 제거**), 안내의 "키워드"는 **클릭 가능한 빨강 필드값**이라 누르면 해당 단계의 응답화면으로 복귀한다. 제출에 성공하면 `participantToken`을 localStorage에 저장하고 `waiting`(대기)으로 넘어간다.
- **waiting(대기, `WaitingScreen`)**: 응답을 보냈다는 안내와 **세로형 스텝퍼** + **응답 마감까지 카운트다운**을 보여준다(`totalParticipants` prop = 참여자 수). **열람 조건**: `ready = allResponded || remaining<=0` — 마감(0초) 전이라도 **전원 응답이 완료되면**(둘 중 먼저) 캘린더를 볼 수 있다. `ready`가 되면 ①②는 완료로 바뀌고 ③이 `캘린더를 확인해보세요`로 **활성화**되며 카운트다운은 사라지고 하단 버튼이 활성화돼 `done`으로 넘어간다(자동 이동 아님). 타이머는 제품 흐름 확인용 **7초 고정**(`WAIT_SECONDS=7`, 실제로는 응답 마감 시각까지의 남은 시간으로 대체 예정).
  - **응답률 표시**: 사람 아이콘 👥 + `{responded}/{total} 응답 완료` + `응답 마감까지 mm:ss`. 데모 시뮬레이션(내 응답 1건에서 시작해 마감 약 2초 전 전원 도달).
  - **애니메이션**: 상단 문구/스텝을 AOS로 순차 등장(`animate-fade-up-blur` + 지연 스태거), 대기 중 활성 스텝 원에 느리고 부드러운 원형 pulse(`.modu-pulse-ring`), 스텝 연결선은 파란 순차 채움(`.modu-line-fill`, 위 스텝이 done 인 구간만·카운트다운 완료 시 다음 구간 이어짐). 스텝은 고정 높이(모바일 `h-[8rem]` / PC `sm:h-[8.5rem]`)라 텍스트가 바뀌어도 간격이 일정하다. 2단계 라벨은 `다른 참여자들의 응답을 기다리고 있어요` → `…받았어요`가 **문장 전체 clip-path 좌→우 와이프 + 페이드**로 전환된다(색 포함).
  - **하단**: `내 응답 수정하기`를 CTA **위**에 배치하고(응답 수정은 이 화면에서 제공), `ready`(완료) 시에는 숨긴다.
- **result(추천 시간 화면, `ResultScreen`)**: 코드는 보존하지만 현재 플로우에서는 비활성화했다. 대기 화면은 `done`으로 바로 이동하고, 예전 임시저장값이 `result`를 들고 있어도 `done`으로 매핑한다. 필요하면 `waiting` → `result` → `done` 동선을 복구할 수 있다.
  - **추천안 목록(맥락형 그룹)**: 1·2·3 순위 번호 나열 대신 **같은 조건의 후보를 그룹 라벨로 묶어** 보여준다(`contextualResult.rankGroups` — 예: `모두 참석할 수 있는 날짜`/`필수참석자가 모두 가능한 날짜`/`차선 날짜`/`피하는 게 좋은 시간`). 그룹 안은 시간순 정렬.
  - **추천안 카드 = 설명 카드**: 선택/투표 대상이 아니다. 시간·등급 배지·이유(누가 참석할 수 있는지)를 보여주는 읽기 전용 카드로, 테두리 없이 밀착 드롭섀도우(`shadow-[0_1px_4px_rgba(15,23,42,0.12)]`)만 쓴다. 클릭 선택·`투표하기`·`투표됨`/`N표` 배지는 모두 제거했다.
- **done(캘린더, `SubmittedCalendarScreenWide`)**: **한 달 달력**(`CalendarGrid`)로 회의 기간을 보여준다. PC 는 달력(좌)+명단 패널(우) 2열 배치이고, 이 화면만 페이지 폭을 넓게 쓴다(`sm:w-[min(80rem,calc(100vw-4rem))]`). 캘린더 날짜칸에는 가능/불가 숫자를 쓰지 않고 **전체 기간 안의 상대 추천도**를 원 3개로 보여준다. 상위권은 파란 원 3개, 중간권은 주황 원 2개, 하위권은 빨간 원 1개를 채우며, 이 등급은 절대값이 아니라 해당 기간의 날짜 점수 순위로 정한다.
  모바일은 달력만 좌우 여백을 최소화(달력 카드에 `-mx-4` 로 본문 `px-4` 를 상쇄 → 화면 폭까지 풀블리드 + 내부 `px-1`)해 정사각(`aspect-square`) 셀을 키우고, 회의 마감일 배지는 좁은 셀을 넘치지 않게 `마감`으로 축약한다(PC 는 `회의 마감일` 전체 라벨·좌우 여백 유지). 캘린더 위에는 맥락형 해석 문장(headline+comment)과 특정 시간대 경고(warnings, 최대 3줄)를 `space-y-2` 간격으로 노출한다. 코멘트·경고에서는 날짜(`N월 N일`)·시간(`HH:MM`/`HH:MM~HH:MM`)·이름(`○○님`) 키워드를 `highlightKeywords`(순수 표시용 정규식)로 **굵게** 강조한다(헤드라인은 이미 굵어 대상 아님). 경고는 후보 슬롯 병합이 아니라 **참석자의 실제 busy 시각**을 쓰고(날짜·시간 검색과 일치 — `buildWarnings`가 스냅샷 블록으로 계산해 `buildContextualScheduleResult`에 `detail`로 전달), **모든 날에 같은 시간대가 반복되면 `매일 …` 한 줄로 묶는다**(시나리오 4의 `recurringBusy`).
  날짜를 선택하면 테두리 링 대신 **셀이 살짝 떠오르는 pop 효과**(`.modu-cell-pop`)가 적용된다. 달력 카드는 테두리 없이(`border-none`) 넓게 쓰되 모바일에서는 그림자를 제거하고 PC에서는 약한 그림자를 유지한다. 우측 명단 패널은 테두리 없이 드롭섀도우를 쓴다. 날짜를 누르면 우측 패널에 참석자별 가능/불가능/미응답 명단과 날짜 전체 요약이 보이며, 그 아래 날짜·시간 검색 카드로 특정 시각을 조회하면 **결과가 이 우측 명단 카드 내용을 대체해** 표시된다(검색 카드에는 입력만, ✕로 지우면 다시 선택 날짜 명단). 다른 날짜를 누르면 검색 결과는 지워지고 그 날짜 명단으로 돌아간다.
  모바일(≤639px)에서는 이 명단 카드를 세로 스택에 두지 않고, 날짜를 탭하면 명단 내용(`rosterContent`)을 아래에서 올라오는 바텀시트 모달(`animate-sheet-up`, `document.body` 포털)로 띄운다. 해석 문장·달력은 그대로 두고 상세만 덮는다. 배경 딤 탭·`Esc`·우측 상단 닫기(✕, `Emoji` 컴포넌트의 TossFace 글리프 — SVG 직접 그리지 않음) 아이콘으로 닫고, 열려 있는 동안 배경 스크롤을 잠근다(`useScrollLock`). 시트는 `max-h-[85dvh]`로 내부 스크롤하므로 예외 시간이 많은 날도 넘치지 않는다. 검색 결과 표시 중에는 패널 자체 ✕(검색 지우기)로 명단으로 돌아온다. 날짜·시간 검색 카드는 캘린더 아래에 그대로 두며, 모바일에서는 그림자를 제거하고 검색하면 모달이 열려 결과를 보여준다. PC(≥640px)는 종전대로 우측 열에 명단 카드를 고정한다.

후보·캘린더 화면은 케이스 기반 데모다:

- 날짜를 눌렀을 때 나오는 우측 날짜 카드의 상단 요약은 모든 상황에서 **`가장 나은 시간`** 기준으로 통일한다. 전부 가능한 날은 `이 날은 모든 시간이 괜찮아요.`만 보여주고, 전원이 가능한 후보 슬롯이 그렇지 않은 슬롯보다 많으면 특정 시간 하나를 추천하기보다 `이 날은 대부분 시간이 괜찮아요. 다만 HH:MM~HH:MM만 피해주세요.`처럼 피할 시간을 먼저 알려준다. 가능한 슬롯이 많지 않을 때는 필수참석자 충돌이 가장 적고 참석 가능 인원이 많은 가장 이른 시간대를 추천한다.

- `done`(캘린더) 화면은 실제 응답이 아니라 **`docs/cases.md`의 케이스 더미 데이터**(`data/demoCases.ts`)로 동작한다. `result`도 같은 데이터로 다시 살릴 수 있게 보존되어 있지만 현재 사용자 플로우에서는 비활성이다. 실제 회의 생성·응답과는 아직 연결하지 않았다.
- 두 화면 모두 **케이스 선택** 과 **케이스 설명(`CaseDescription`: 상황·솔루션 + info/caution/danger 경고 배너)** 을 제공한다. 선택한 `caseId`는 `ResponseForm`에서 두 화면이 공유한다.
- **PC·모바일 공통으로 드롭다운 모달(`FloatingCaseSelector`)** 을 쓴다 — `상황 {N}` 버튼을 탭하면 케이스 칩 + `CaseDescription` 패널이 그 아래로 열리고, 바깥 탭/✕/Esc 로 닫힌다. 현재 활성 화면인 `done` 헤더의 **`모두의 날짜` 제목 바로 오른쪽**에 조용한 슬레이트 칩을 붙인다(응답 알약·`추천안 보기` 아이콘은 헤더 우측). 드롭다운은 화면 밖으로 넘치지 않게 `posFromRect`로 좌우 16px 여백 안에 맞춘다. (구버전 데스크톱 인라인 `CaseSelector`(호버 펼침)·done 우하단 플로팅 배치는 **삭제** — git 히스토리 참고.)
- 추천안 카드(`ResultScreen`): **순위 번호 대신 같은 조건의 후보를 그룹으로 묶어**(`contextualResult`의 `rankGroups` — 라벨 예: `모두 참석할 수 있는 날짜`/`필수참석자가 모두 가능한 날짜`/`차선 날짜`/`피하는 게 좋은 시간`) 그룹 라벨 + 카드 목록으로 보여준다. 카드에는 시간(`formatKoreanDateTimeRange`) · 등급 배지(`GRADE_LABELS`, caution 은 레드 계열) · 이유. 카드는 **설명용 읽기 전용**이며 선택·투표 인터랙션이 없다(§4.4 result 참고).
- `buildCaseCandidates`가 우선순위(① 전원 가능 ② 필수 더 많이 ③ 선택 더 많이 ④ 이른 날짜·시간)로 정렬하고 등급(`best`/`recommended`/`conditional`/`caution`)을 도출한다(하드코딩 아님). `buildCaseSnapshot`이 같은 슬롯에서 캘린더용 `busy` 블록·참석자 응답 상태를 파생한다. 회의 기간이 길면 `buildCaseSlotsForDates`가 케이스의 불가 패턴을 기간 전체에 분산한다.

수정 UX:

- 첫 제출 후 `participantToken`을 localStorage에 저장한다.
- 같은 브라우저로 다시 접속하면 본인확인 없이 곧장 `waiting`(대기 화면)으로 진입하고(매 방문마다 카운트다운을 거쳐 `done`으로 이동), 대기 화면의 `내 응답 수정하기`를 누르면 가능 시간 입력을 처음부터 다시 받는다. (새 입력 모델이라 기존 블록을 역으로 채우지는 않는다.)
- 토큰이 없거나 일치하지 않으면 회의 안내(intro)부터 다시 진행한다.

### 4.5 참석자 결과 화면 (설명 가능한 결정 화면)

별도 관리자 페이지는 두지 않는다. 회의 생성자도 참석자 중 한 명으로 `/m/[meetingId]` 화면에서 응답 현황, 전체 캘린더, modu 의 판단 근거를 확인한다. **투표 UI 는 없다** — 결과 캘린더는 "어디에 투표하지?"가 아니라 "어느 날을 먼저 보고 어느 날을 피하면 좋지?"를 이해시키는 화면이다.

주요 컴포넌트:

- `ResponseForm` 내부 `WaitingScreen`
- `ResponseForm` 내부 `ResultScreen`(비활성 보존)
- `ResponseForm` 내부 `SubmittedCalendarScreenWide`

캘린더 정보 위계:

1. 상황 해석 문장(headline/comment)
2. 날짜 셀의 상대 추천도 원 3개
3. 피하면 좋은 날과 낮은 추천도 신호
4. 날짜 전체 요약과 예외 시간대
5. 특정 날짜·시간 조회 결과의 가능/불가능/미응답 명단

결정 규칙:

- 회의 생성 시 정한 필수/선택 참석 여부는 결과 화면에서 확인만 한다.
- 응답 이후 누군가가 필수/선택 참석 여부를 바꿔 결과 해석을 뒤집는 흐름은 제공하지 않는다.
- modu는 회의 시간을 확정하지 않는다. 결과 캘린더는 판단 근거를 보여줄 뿐이고, 최종 시간은 참여자들이 제품 밖(Slack·카톡·구두 등)에서 정한다. 결과 캘린더 하단에도 참여자들과 최종 시간을 정해보라는 안내가 노출된다.
- 필수참석자가 모두 가능한 후보가 없으면 기간 조정을 안내한다. 비슷한 후보는 같은 그룹으로 묶고 이른 시간을 먼저 보여준다.
- 확정/수동 확정 버튼은 제공하지 않는다. `/meetings/[id]/confirmed`와 `.ics`는 legacy 호환용으로만 남는다(신규 플로우 진입 없음).

### 4.6 확정 화면

주요 컴포넌트:

- `ConfirmedMeetingSummary`(공유용 문구 텍스트와 `.ics` 다운로드 버튼을 컴포넌트 내부에 인라인 포함 — 별도 `ShareSummaryText`/`IcsDownloadButton` 없음)

표시 정보:

- 회의 길이
- 참석 대상 요약
- 필수 참석자 충족 여부
- 공유용 문구
- `.ics` 다운로드 버튼

## 5. UI 스타일 방향

modu는 한국어 협업 도구처럼 간결하고 신뢰감 있게 만든다.

디자인 원칙:

- 넓은 여백
- 명확한 카드 구조
- 큰 타이포그래피
- 부드러운 라운드
- 쉬운 문장
- 모바일/데스크톱 반응형
- 색상에만 의존하지 않는 상태 표현
- 페이지 전체·헤더 배경은 `rgb(255 255 255 / 0.95)`(Tailwind `bg-white/95`)로 통일한다. 단 **모바일 고정 헤더는 불투명 흰색(`bg-white`) + 하단 테두리 없음**으로 하고, 헤더 바로 아래를 흰색→투명 그라데이션(`.modu-mobile-header::after`)으로 흐려 본문과 부드럽게 잇는다. 데스크톱(헤더 static)에서는 그라데이션을 끈다.
- 모달은 화면 목적을 나타내는 타이틀만 두고, 불필요한 서브타이틀이나 설명 문구를 넣지 않는다.
- 모달이 열려 있는 동안 배경(body) 스크롤을 잠근다(`lib/useScrollLock`, iOS Safari 까지 막히도록 `position: fixed` + 스크롤 위치 복원 방식). 닫기 버튼은 충분한 터치 영역(`h-11 w-11`, 아이콘 20)으로 둔다.

토스 제품처럼 쉽고 명확한 인상을 참고하되, 토스 브랜드, 로고, 고유한 시각 요소를 직접 모방하지 않는다.

결과 캘린더가 가장 중요한 UI다. 화면 전체가 날짜별 상대 추천도와 판단 근거를 빠르게 훑고, 필요한 날짜·시간을 바로 조회하는 데 집중되어야 한다.

## 7. 접근성 및 반응형 기준

- 모든 버튼은 명확한 텍스트 또는 접근 가능한 라벨을 가진다.
- 상태 색상에는 텍스트 라벨을 함께 붙인다.
- 시간표 셀은 키보드 포커스가 가능해야 한다.
- 모바일에서 텍스트가 버튼 밖으로 넘치지 않아야 한다.
- 모바일에서 `input`, `textarea`, `select`를 터치했을 때 iOS Safari 또는 Chrome이 화면을 자동 확대하지 않아야 한다. 입력 필드의 실제 `font-size`는 모바일에서도 16px 이상으로 유지하고, focus 시 강제 `scrollIntoView` 호출이나 viewport scale 조작을 넣지 않는다. 키보드는 올라오되 화면 확대나 입력 영역 잘림이 없어야 한다.
- 캘린더 셀의 날짜, 마감 배지, 상대 추천도 원 3개는 작은 화면에서도 겹치지 않고 먼저 보여야 한다.
- 모바일 주요 CTA 버튼(회의 만들기, 응답 제출 등)은 `MobileStickyAction`을 사용한다. 하단 영역은 테두리/그림자/블러 없이 본문 배경과 같은 불투명 배경을 유지하고, 본문과 맞닿는 위쪽 경계만 그라데이션으로 자연스럽게 이어지게 한다.
- 모바일 주요 CTA 때문에 짧은 화면(flex 레이아웃)에 가짜 스크롤이 생기면 안 된다. 본문이 길어 하단 고정 영역에 가릴 때만 스크롤되며, 이때도 마지막 내용과 하단 영역 사이에 일정한 여백을 둔다(스페이서 동작은 §2 참고).
- 모바일에서 `다음` 버튼으로 다음 단계로 넘어갈 때 다음 입력에 **자동 포커스하지 않는다**(키보드가 자동으로 올라오며 본문이 밀려 올라가는 현상 방지). 입력은 사용자가 직접 터치할 때만 활성화하고, 상단 문장이 갱신될 때 본문을 자동으로 스크롤하지 않는다. 데스크톱은 기존대로 `다음` 시 자동 포커스한다. (회의 만들기·참석자 응답 공통)
- 참석자 응답 화면의 모든 단계(intro/본인확인/가능 시간) 하단 CTA도 회의 만들기 화면과 같이 `MobileStickyAction`으로 모바일 하단에 고정한다.
- 복사 버튼, 확정 버튼 등 주요 액션은 터치하기 충분한 크기로 만든다.

## 8. 프론트엔드 테스트 시나리오

- 랜딩에서 회의 만들기로 이동할 수 있다.
- 랜딩은 4장면 인트로 모션(수집→해석→선별→후보 리스트)이 자동 재생되고, 재생이 다 끝나야 `회의 만들기` CTA가 하단 고정으로 페이드인된다(건너뛰기·다시 보기 없음). 새로고침하면 다시 처음부터 재생되며, 저감 모션만 마지막 장면 + CTA를 바로 보여준다.
- 회의 생성 폼에 기본값이 올바르게 들어간다.
- 회의명 20글자, 안건 30글자, 장소 20글자를 넘으면 라벨이 아이콘 포함 경고 문구로 바뀌고 입력창 테두리가 빨간색으로 바뀐다.
- 가상 직원 목록에서 참석자를 검색하고 선택/해제할 수 있다.
- 참석자는 최대 8명까지만 선택할 수 있다.
- 참석자를 1명 이하로 둔 채 `선택 완료`를 누르면 모달이 닫히지 않고 유효성 경고가 보인다.
- 선택된 참석자 뱃지를 필수참석/선택참석 구역으로 드래그해 참석유형을 변경할 수 있다.
- 공유 화면에서 참석자 링크를 복사할 수 있다.
- 참석자 링크 진입 시 회의 안내(intro) → `시간 정하러 가기` → 본인확인(이름·직무, 오류 시 토스트, 성공 시 `확인 완료` 토스트) 순으로 진행한다.
- 달력·시간 선택 등 모달이 열려 있는 동안 배경(본문)이 스크롤되지 않는다.
- 가능 시간을 불가 중심 2단계(불가능한 날짜 → 특정 날짜+시간)로 입력하고, `다음` → 입력 확인(review) → `다음`으로 제출할 수 있다. 날짜는 달력에서 다중 선택하고, 특정 날짜의 안 되는 시간대는 여러 개 추가한다.
- 불가능한 날짜는 `날짜 선택` 버튼 → 달력 모달에서 다중 선택하며, 한 달씩 표시되고 `<` `>`로 월을 이동하고, 토·일은 선택할 수 없다. 특정 날짜+시간 모달에서는 불가능으로 고른 날짜가 `불가` 배지로 비활성화된다.
- 특정 날짜의 안 되는 시간은 근무시간 밖이면 회색 + `근무시간 외`로 표시되고 컨펌을 거쳐 선택할 수 있으며, 같은 날 동일 시간대는 중복 추가되지 않는다.
- 추가한 '특정 날짜+시간'은 빨간 뱃지(칩) 목록으로 쌓인다(모바일 모달 안·페이지 요약 공통, `flex flex-wrap`). 추가 시 `fade-in`, ✕ 삭제 시 `fade-out` 후 제거되며, 모바일 전체화면 달력 모달은 아래에서 위로 슬라이드로 열리고(`sheet-up`) 닫힐 때 페이드아웃한다.
- 같은 브라우저에서 제출 후 다시 접속하면 본인확인 없이 대기 화면(waiting)으로 바로 진입하고(카운트다운 후 done 캘린더), `내 응답 수정하기`로 다시 입력할 수 있다.
- 대기 화면은 7초 카운트다운을 두되 전원 응답이 먼저 완료되면 조기 열람되며, 응답률(👥 N/M)과 마감 카운트다운을 표시한다.
- 제출 후 캘린더 화면(done)에서 케이스를 골라 해석 문장과 날짜별 상대 추천도를 본다. 케이스 선택은 PC·모바일 공통으로 `모두의 날짜` 제목 옆 `상황` 버튼을 탭하면 드롭다운 모달이 열린다.
- 달력 날짜칸에는 전체 기간 안의 상대 추천도 원 3개가 표시되고, 캘린더 위에 결과 분포별 해석 문장(대부분 가능/보통/바쁨/없음)이 나온다. 날짜를 누르면 셀이 살짝 떠오르고(pop), 아래 패널에서 참석자별 가능/불가능/미응답 명단을 볼 수 있다. 상세 메모 원문은 보이지 않는다.
- 참석자 결과 화면에서 응답자와 미응답자를 구분해 볼 수 있다.
- 참석자 결과 화면에서 전체 캘린더로 불가능 시간 집계와 날짜별 상대 추천도를 볼 수 있다.
- 날짜를 누르면 날짜 전체 요약과 예외 시간대를 볼 수 있고, 날짜·시간 조회로 특정 시각의 가능 여부를 확인할 수 있다.
- 응답 이후 필수/선택 참석 여부를 임의로 변경할 수 없다.
- 모든 참석자가 응답하기 전에는 잠정 결과만 보여준다.
- 전원 응답 또는 열람 조건 충족 시 modu가 해석 문장과 캘린더 신호를 정리해 보여준다. 회의 시간을 확정하지는 않으며, 최종 결정은 참여자들이 외부에서 한다.
- legacy 확정 화면에서 공유 문구 복사와 `.ics` 다운로드가 가능하다.

### 9.2 두 개의 "문장 빌더" 상태 머신

두 큰 폼은 같은 패턴(상단 = 쌓이는 한국어 문장, 하단 = 현재 단계 입력 + CTA)을 쓴다.

- **`MeetingCreateForm`** — 단일 `step`(0 회의명 → 1 안건 → 2 장소 → 3 진행시간 → 4 회의마감일 → 5 응답마감일 → 6 참석자). hidden input으로 전체 값을 항상 직렬화해 Server Action에 제출. 마지막 단계 뒤 `confirming` 상태로 **회의 확인 화면**(읽기 문장 + 클릭 키워드 수정 + 글자 단위 잉크 채움)으로 전환하고, 7번 문구 등장 후(`confirmCtaReady` 타이머) 노출되는 `회의 만들기`(`SubmitButton`)가 실제 제출한다.
- **`ResponseForm`** — 다단계 머신:
  - `step`: `loading | intro | identity | availability | review | waiting | result | done`
  - `identity`의 `formStep`: 0 이름 → 1 직무(Select)
  - `availability`의 `availStep`(불가 중심 2단계): 0 불가능한 날짜 → 1 특정 날짜+시간 (`maxAvailStep`로 도달 단계 추적). `buildBlocks`는 `busy`만 저장.
  - `review`(입력 최종 확인, `다음`으로 `submitAvailability` 제출) → `waiting` → `done`(`SubmittedCalendarScreenWide` 월간 캘린더). `result`(`ResultScreen`)는 타입과 컴포넌트에 남아 있지만 현재 플로우에서는 `null` 처리/우회한다.
  - `done`은 **케이스 더미**(`data/demoCases.ts`)로 동작 — 공유 `caseId`로 `FloatingCaseSelector`(케이스 선택 모달)/`CaseDescription`/캘린더를 그린다(실데이터 아님).
  - 복귀(localStorage 토큰) 사용자는 `waiting`(대기)으로 진입한다(카운트다운 후 `done`).

### 9.4 핵심 규칙·상수

- 시간은 전부 **KST(고정 +09:00)**, 근무시간 **09:00~18:00 고정**, 슬롯/셀 **30분**.
- 모든 사용자 노출 문구는 **`docs/ux-writing.md`**(해요체·능동·긍정·캐주얼 경어·명사 조합 금지)를 따른다. 동적 값 뒤 어미는 `lib/korean.ts`의 `hasBatchim()`으로 분기.
- 상태색에는 **항상 텍스트 라벨**을 함께 붙인다(색만으로 상태 전달 금지).
- 모든 Supabase 접근은 **서버 전용**(`server-only`), service_role로 RLS 우회 → 참석자 수정 권한은 앱 내부 **토큰 비교**(`participant_token`)에 의존한다. 제품 용어에 관리자 개념은 없다.
- 입력 한도는 `lib/meetingLimits.ts`(제목 20 / 안건 30 / 장소 20자, 참석자 2~8명).

### 9.5 함정 / 주의

- `ResponseForm.tsx`(2300+줄)·`MeetingCreateForm.tsx`(800+줄)는 한 파일에 하위 컴포넌트 다수. 수정 전 파일 내 정의 위치부터 grep.
- 응답 폼의 인라인 `MonthCalendar`/`PickerSheet`는 제거되고 공용 `components/scheduler/CalendarModal.tsx`(내부 `CalendarGrid`)로 분리됐다. 달력 UI 수정은 이 파일에서.
- 대기 화면(`WaitingScreen`) 카운트다운은 제품 흐름 placeholder라 **7초 고정**(`WAIT_SECONDS=7`)이며 마감 전 전원 응답 시 조기 열람된다. 실 서비스는 응답 마감 시각까지의 남은 시간·실제 응답률로 대체 예정.
- 스텝퍼 애니메이션(`.modu-pulse-ring`/`.modu-line-fill`)은 **`app/globals.css`로 이동**했다(개발 중 `tailwind.config.ts` 변경이 HMR에 미반영되는 문제 회피 + `prefers-reduced-motion` 대응). 두 클래스를 고칠 때 config 가 아니라 globals.css 를 본다.
- 가능 시간 `buildBlocks`는 불가(`busy`)만 저장하고 **근무시간 행만** 셀로 환산 → 완전히 근무시간 밖인 시간대는 저장 시 무시됨(§4.4 참고).
- 제출 후 활성 화면인 `done`(`SubmittedCalendarScreenWide`)은 **케이스 더미**(`data/demoCases.ts`, `buildCaseCandidates`/`buildCaseSnapshot`)로 동작한다(실제 응답 아님). 비활성 보존 중인 `result`(`ResultScreen`)도 같은 모델을 쓴다. 화면 색·문구는 **맥락형 해석 레이어**(`lib/scheduler/contextualResult.ts` + `data/demoCaseAdapter.ts`)가 만든 결과 모델을 쓴다 — 실데이터 연결 시 `toSchedulerInput → evaluateAllSlots → buildContextualScheduleResult` 로 같은 모델을 만들면 된다. 파일에 남은 `buildDummySnapshot`/`DUMMY_PEOPLE`은 더 이상 호출되지 않는 잔여 코드다.
- Supabase env 없으면 `lib/demoMeeting.ts` **데모 플로우**로 빠진다(제출 미저장·확정 불가). 로컬 개발은 `.modu-local-db.json` 파일 폴백.
- 점심 비활성화는 `lunch_start='00:00'/lunch_end='00:01'` 센티넬로 저장(서버가 점심 겹침 블록을 거부).
- `ExpiryNotice.tsx` 컴포넌트는 남아 있으나 어느 화면에서도 쓰지 않는다(만료/삭제 안내는 노출하지 않음).
- 캘린더(done) 화면 래퍼는 와이드 브레이크아웃을 위해 `sm:-translate-x-1/2`(transform)를 쓰는데, **`transform` 조상 안의 `position:fixed` 자식은 뷰포트가 아니라 그 조상 기준으로 갇힌다.** 그래서 이 화면에 전체 화면 딤/오버레이(`fixed inset-0`)를 그대로 두면 본문 폭만 어두워진다. `FloatingCaseSelector`(상황 선택)는 backdrop+dialog 를 `createPortal(…, document.body)`로 빼고 dialog 는 트리거 rect 기준 `fixed`로 배치해 해결했다. 이 화면에 새 오버레이를 붙일 때도 같은 포털 패턴을 쓴다.
