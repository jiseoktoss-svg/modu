# modu 프론트엔드 구현 문서

## 1. 프론트엔드 목표

modu의 프론트엔드는 사용자가 30초 안에 서비스 가치를 이해하고, 복잡한 일정 조율 과정을 부담 없이 완료할 수 있도록 구성한다.

핵심 UX 목표:

- 주최자는 빠르게 회의를 만들고 링크를 복사할 수 있다.
- 참석자는 로그인 없이 모바일에서도 쉽게 시간표를 입력할 수 있다.
- 참석자는 모든 응답이 모인 뒤 후보 시간대에 투표할 수 있다.
- 주최자는 전체 캘린더, 추천 후보의 이유, 투표 현황을 한눈에 이해할 수 있다.
- 모든 상태는 색상뿐 아니라 텍스트 라벨로도 표현한다.

## 2. 기술 기준

- Next.js App Router
- TypeScript
- Tailwind CSS
- Server Components 기본 사용
- 상호작용이 필요한 폼, 시간표, 복사 버튼, 추천 재계산 UI는 Client Components로 분리
- 아이콘이 필요한 버튼에는 `lucide-react` 사용
- 이모지는 `public/fonts/TossFaceFontMac.ttf`를 `TossFace` 폰트로 등록해 렌더링

## 4. 화면별 구현 상세

### 4.1 랜딩 화면

필수 콘텐츠:

- 서비스명: `modu`
- 메인 카피: `6명의 조건을 비교해, 모두가 납득할 수 있는 1시간을 찾아요.`
- CTA: `회의 만들기`
- 보조 CTA: `샘플 회의 체험하기`
- 짧은 가치 설명:
  - 필수 참석자와 선택 참석자를 구분한다.
  - 선호 날짜, 불가능 날짜, 날짜별 시간대 조정을 함께 고려한다.
  - 추천 이유를 한국어로 설명한다.
- 개인정보 마이크로카피:
  - `상세 일정명은 저장하지 않고, 가능 여부와 선호 상태만 사용해요.`

화면 방향:

- 첫 화면에서 제품명과 핵심 카피가 분명히 보여야 한다.
- 랜딩만 읽어도 단순 투표 앱이 아니라 의사결정 도구라는 점이 드러나야 한다.
- 마케팅 페이지처럼 과하게 장식하지 않고, 실제 서비스 진입을 우선한다.

### 4.2 회의 만들기 화면

주요 컴포넌트:

- `MeetingCreateForm`
- `ParticipantListEditor`

폼 기본값:

- 회의 길이: 1시간 0분
- 회의 마감 날짜: 오늘 기준 7일 뒤
- 내부 계산용 근무 시간: 09:00~18:00
- 신규 회의의 날짜 범위: 오늘부터 회의 마감 날짜까지
- 참석자: 6명 예시

입력 항목:

- 회의명
- 안건
- 장소
- 회의 마감 날짜
- 회의 길이(시간, 분 직접 입력)
- 참석자 모달(가상 직원 검색/선택, 필수/선택)

검증:

- 회의명은 필수다.
- 참석자는 최소 2명 이상이어야 한다.
- 회의 마감 날짜는 필수다.
- 회의 길이는 시간과 분을 직접 입력한다.

주최자는 근무 시작/종료 시간과 점심 시간을 입력하지 않는다. 화면에서는 회의 마감 날짜만 받고, 서버가 오늘부터 마감 날짜까지의 범위와 09:00~18:00 고정 근무 시간을 저장한다.

참석자 모달:

- 50명의 가상 직원 목록을 보여준다.
- 직원은 이름과 직책을 가진다.
- 이름 또는 직책으로 검색할 수 있다.
- 직원 목록에서는 참석자를 선택하거나 선택 해제한다.
- 선택된 참석자 목록에서는 참석유형만 `필수` 또는 `선택`으로 변경할 수 있다.
- 모달 상단에는 `참석자 선택` 타이틀만 두고, 별도 서브타이틀이나 설명 문구는 두지 않는다.

### 4.4 참석자 응답 화면

주요 컴포넌트:

- 날짜 카드 UI
- 날짜별 시간 편집 모달
- 하단 고정 제출 버튼
- 제출 후 회의 캘린더 화면
- `VotingPanel`

날짜 상태:

- `available`: 가능
- `busy`: 불가능
- `preferred`: 선호

응답 방식:

- 오늘부터 회의 마감 날짜까지 날짜 카드를 보여준다.
- 각 날짜는 전체 `가능`, `선호`, `불가능`으로 빠르게 지정할 수 있다.
- 각 날짜의 `시간 편집`에서 09:00~18:00 범위를 30분 단위로 조정할 수 있다.
- 시간 편집은 상태를 먼저 고르고 시간대를 탭하는 방식으로 동작한다.
- 선택하지 않은 시간대는 기본적으로 `가능`으로 본다.

중요한 데이터 정책:

- `available`은 기본 상태이며 DB에 저장하지 않는다.
- 새 응답 플로우에서는 변경된 `preferred`, `busy` 시간 블록만 서버에 제출한다.
- 기존 데이터 호환을 위해 서버와 추천 알고리즘은 `avoid` 상태도 읽을 수 있다.

투표 UX:

- 응답 제출 후 같은 브라우저에서는 `participantToken`으로 본인 응답을 수정할 수 있다.
- 응답 제출 후에는 단순 완료 화면이 아니라 시간축이 있는 회의 캘린더 화면을 보여준다.
- 회의 캘린더는 날짜 컬럼과 30분 단위 시간 행으로 구성하고, 각 칸에 가능/선호/불가능/미응답 집계를 표시한다.
- 회의 캘린더는 넓은 화면에서 실제 캘린더 서비스처럼 시간 축과 날짜 컬럼이 한 화면에 보이도록 구성한다.
- 캘린더 칸을 선택하면 해당 시간대의 참석자별 상태 요약을 볼 수 있다. 상세 사유와 메모 원문은 표시하지 않는다.
- 캘린더에는 새로고침과 응답 수정 액션을 제공한다.
- 모든 참석자가 응답하기 전에는 후보 투표 대신 대기 안내를 보여준다.
- 모든 참석자가 응답하면 추천 후보 목록을 보여주고, 참석자는 후보 중 하나에 투표한다.
- 같은 참석자가 다시 투표하면 기존 투표를 새 후보로 교체한다.

모바일 UX:

- 각 셀의 터치 영역은 충분히 커야 한다.
- 상태 선택은 세그먼트 컨트롤 또는 하단 고정 툴바로 제공한다.
- 드래그 선택이 어렵다면 알파에서는 "상태 선택 후 셀 탭" 방식으로 구현한다.
- 날짜가 많을 경우 날짜 카드는 2열 또는 1열 반응형 그리드로 표시하고, 시간 편집은 모달에서 처리한다.
- 제출 후 캘린더는 모바일에서 가로 스크롤을 허용하되, 시간 축은 고정해 현재 행의 의미를 잃지 않게 한다.

수정 UX:

- 첫 제출 후 `participantToken`을 localStorage에 저장한다.
- 같은 브라우저에서 다시 접속하면 기존 응답을 불러와 수정 가능 상태로 보여준다.
- 토큰이 없거나 일치하지 않으면 새 응답 또는 본인 선택 흐름으로 유도한다.
- 이미 제출한 참석자가 같은 브라우저로 접속하면 완료 카드가 아니라 회의 캘린더 화면을 바로 보여준다.

- 1위가 동률이면 동률 후보들에 확정 버튼을 보여주고, 주최자가 하나를 골라 확정한다.
- 후보 확정 전 확인 다이얼로그를 보여준다.

## 5. UI 스타일 방향

modu는 한국어 협업 도구처럼 간결하고 신뢰감 있게 만든다.

디자인 원칙:

- 넓은 여백
- 명확한 카드 구조
- 큰 타이포그래피
- 부드러운 라운드
- 쉬운 문장
- 모바일/데스크톱 반응형
- 색상에만 의존하지 않는 상태 표현
- 모달은 화면 목적을 나타내는 타이틀만 두고, 불필요한 서브타이틀이나 설명 문구를 넣지 않는다.

토스 제품처럼 쉽고 명확한 인상을 참고하되, 토스 브랜드, 로고, 고유한 시각 요소를 직접 모방하지 않는다.

추천 카드가 가장 중요한 UI다. 화면 전체가 추천 후보를 빠르게 비교하는 데 집중되어야 한다.

## 6. 컴포넌트 구조

권장 구조:

```txt
components/
  meeting/
    MeetingCreateForm.tsx
    ParticipantListEditor.tsx
    ShareLinksPanel.tsx
    ConfirmedMeetingSummary.tsx
  scheduler/
    ResponseForm.tsx      # 응답 입력, 제출 후 캘린더, 후보 투표
    AvailabilitySummaryCalendar.tsx
    RecommendationCard.tsx
    RecommendationList.tsx
  ui/
    Button.tsx
    Card.tsx
    Badge.tsx
    Input.tsx
    Select.tsx
    Textarea.tsx
```

구현 원칙:

- 서버 데이터 조회는 가능한 Server Components에서 처리한다.
- 브라우저 상태가 필요한 컴포넌트만 Client Components로 만든다.
- 폼 제출은 Server Actions를 우선 사용한다.
- Route Handler는 `.ics` 다운로드처럼 파일 응답이 필요한 경우에 사용한다.

## 7. 접근성 및 반응형 기준

- 모든 버튼은 명확한 텍스트 또는 접근 가능한 라벨을 가진다.
- 상태 색상에는 텍스트 라벨을 함께 붙인다.
- 시간표 셀은 키보드 포커스가 가능해야 한다.
- 모바일에서 텍스트가 버튼 밖으로 넘치지 않아야 한다.
- 추천 카드의 핵심 정보는 작은 화면에서도 접히지 않고 먼저 보여야 한다.
- 복사 버튼, 확정 버튼 등 주요 액션은 터치하기 충분한 크기로 만든다.

## 8. 프론트엔드 테스트 시나리오

- 랜딩에서 회의 만들기와 샘플 체험으로 이동할 수 있다.
- 회의 생성 폼에 기본값이 올바르게 들어간다.
- 가상 직원 목록에서 참석자를 검색하고 선택/해제할 수 있다.
- 선택된 참석자의 참석유형을 변경할 수 있다.
- 공유 화면에서 참석자 링크를 복사할 수 있다.
- 참석자 화면에서 날짜 상태와 날짜별 시간 상태를 변경하고 제출할 수 있다.
- 같은 브라우저에서 제출 후 수정할 수 있다.
- 제출 후 참석자 화면에서 시간축 캘린더가 표시된다.
- 참석자 캘린더에서 특정 칸을 선택하면 참석자별 상태 요약을 볼 수 있고, 상세 메모 원문은 보이지 않는다.
- admin 화면에서 응답자와 미응답자를 구분해 볼 수 있다.
- admin 화면에서 전체 캘린더로 선호/불가능 집계를 볼 수 있다.
- 추천 후보 카드가 최소 3개 표시된다.
- 필수/선택 변경 시 추천 결과가 다시 계산된다.
- 모든 참석자가 응답하기 전에는 후보 투표가 열리지 않는다.
- 모든 참석자가 후보에 투표하면 최다 득표 후보를 확정할 수 있다. 1위가 동률이면 주최자가 동률 후보 중 하나를 고른다.
- 확정 화면에서 공유 문구 복사와 `.ics` 다운로드가 가능하다.
